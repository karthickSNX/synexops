01 What is Apache Airflow?
The orchestrator. Not the compute engine. Understand the difference before you write a single task.
🪂 The one-sentence definition that actually sticks

Airflow is a workflow orchestrator — it tells other tools when to run, in what order, and what to do when they fail. It does not move data. It does not transform data. It coordinates the tools that do.

Created at Airbnb in 2014, the core philosophy is configuration as code: workflows are Python files checked into Git, not clicks in a GUI. That makes them testable, reviewable, and as flexible as Python itself.

🎼 The Conductor Analogy — keep this one forever. Airflow is the conductor, not the musician. The conductor doesn't play any instrument. They coordinate when each section enters, stop the performance if someone falls apart, and restart the brass if the woodwinds weren't ready. Airflow coordinates your data tools. It never touches your data — it ensures the right tools run in the right order with the right inputs.
The concrete problem it solves

You have five pipeline steps wired with cron. Step 3 usually takes 40 minutes — today it took 90. Step 4 fired on schedule before step 3 finished. dbt ran on stale data. Reports went out wrong. Nobody noticed until 10 AM.

With Airflow: step 4 has a declared dependency on step 3. The scheduler waits until step 3 is marked success — period. Step 3 takes 90 minutes? Step 4 starts 90 minutes later. Step 3 fails? The chain stops, retries fire, you get paged. The business never sees wrong data.

Airflow IS built for
  • Batch ETL — daily/hourly extract → transform → load sequences
  • dbt orchestration — schedule, sequence, retry dbt model runs
  • ML training pipelines — feature prep → train → evaluate → deploy
  • Multi-system coordination — Spark, BigQuery, Snowflake, REST APIs
  • Historical backfills — replay any date range with one command
Airflow is NOT for
  • Streaming — minimum granularity is minutes. Use Kafka + Flink.
  • Data processing — it runs Spark; it doesn't replace it
  • Real-time triggers — sub-second latency needs an event bus

🐳 Local-to-prod parity — the only setup that matters

The Docker Compose setup mirrors a real CeleryExecutor deployment: scheduler, webserver, worker, Postgres, Redis. This is how you debug. This is how you catch issues before they hit prod. airflow standalone exists for your first 5 minutes — then you move here and never look back.

02 Architecture
Five components. Each fails differently. Know where the money goes.
🏗 The diagnostic mental model

When something breaks at 2am, the first question is always: which component is sick? Scheduler, Executor, Webserver, Metadata DB, and Workers each have completely different failure signatures. Misidentifying costs hours. This section gives you the model to answer in 30 seconds.

Component overview — with the cost column
Component What it owns Cost profile Failure symptom
Scheduler Parses DAG files every 30s. Decides what runs and when. Sends tasks to executor. High CPU. Every DAG file is imported every 30s. Module-level I/O here destroys CPU. Tasks stop being created. DAG runs fall behind schedule.
Workers Actually execute task code. Process per task (Celery) or pod per task (K8s). High Memory. This is where OOMs happen. Pandas DataFrames, large model loads, memory leaks in task callables. Tasks silently disappear with no log output — SIGKILL leaves no Python trace.
Metadata DB Stores every task state, DAG run, connection, variable, XCom, and log pointer. High IOPS. Scheduler writes a state transition per task state change. At scale this is your first bottleneck, not workers. Scheduler slows. UI becomes sluggish. Tasks queue but never start.
Webserver Flask app. Reads from DB to render UI. Trigger runs, clear tasks, manage connections. Low–medium. Read-mostly from DB. Scales independently. UI down but pipelines still run. DB queries slow = UI slow.
Executor Plugin inside scheduler. Routes tasks to execution environment. Inherited from chosen executor (Celery broker, K8s API server). Tasks queue indefinitely. Broker or K8s API unreachable.
Full execution path — scheduled → done
📁 Scheduler parses dags/my_dag.py every 30s
↓ schedule interval elapsed + all upstreams success
📋 Task instance created → state: "scheduled"
↓ executor picks up the task instance
📤 Placed on queue (Redis / K8s API / subprocess)
↓ state: "queued" → worker picks it
⚙ Worker executes task code → state: "running"
↓ function returns without raising
✅ State written to DB: "success" → downstream unblocked

SymptomRoot cause

The OOM debugging path every senior knows: Task disappears with no log output. No Python exception. No stack trace. You look in Airflow logs — nothing. This is SIGKILL from the kernel OOM killer. The evidence is in worker system metrics (dmesg, CloudWatch, Datadog) not Airflow. Fix: increase worker memory, or reduce task footprint by streaming data instead of loading full DataFrames.

03 DAGs & Tasks
Two gotchas that burn every engineer. Learn them now, not at 3am.
⚡ Gotcha #1 — Module-level code runs in the scheduler

A DAG file is a Python script that Airflow imports. Every Python statement outside a function runs inside the scheduler process every 30 seconds. A database connection, HTTP call, or expensive import at module level = you are DoS'ing your own scheduler. Every database query, API call, and slow operation belongs inside a task callable — never at module level.

⚡ Gotcha #2 — The Data Window is backwards

A @daily DAG fires at midnight January 2nd. Its {{ ds }} is 2024-01-01. It processes yesterday's completed data — not today's in-progress data. This is intentional. Never use datetime.now() in tasks. Backfill 30 days and every run gets today's date. Use {{ ds }} which correctly reflects each run's data window.

The "Data Window" mental model — rebrand this in your team's vocabulary

Stop saying "execution date." Stop saying "logical date." Say Data Window. When you see 2024-01-01, say out loud: "I am looking at January 1st's data." That mental shift prevents 90% of off-by-one errors in SQL queries.

⚠ execution_date still appears in old code and docs everywhere. It's deprecated. It equals logical_date which equals data_interval_start. They're the same thing with different names across Airflow versions. The important one: it's the start of the window the run covers.
DAG: schedule="@daily"
Jan 1 00:00 ── data window opens
January 1st data accumulates...
Jan 2 00:00 ── Run fires 🔵
{{ ds }} = "2024-01-01"
{{ ds_nodash }} = "20240101"
data_interval_start = 2024-01-01T00:00
data_interval_end = 2024-01-02T00:00
Run on Jan 2. Processing Jan 1's data. That's correct.
Production-quality annotated DAG
from airflow.decorators import dag, task from airflow.operators.bash import BashOperator from datetime import datetime, timedelta default_args = { 'owner': 'data-engineering', 'retries': 2, 'retry_delay': timedelta(minutes=5), 'email_on_failure': True, 'email': ['de-oncall@company.com'], } @dag( dag_id='daily_sales_pipeline', start_date=datetime(2024, 1, 1), # NEVER datetime.now() schedule='0 6 * * *', catchup=False, # ALWAYS explicit max_active_runs=1, dagrun_timeout=timedelta(hours=2), default_args=default_args, tags=['sales', 'daily', 'critical'], doc_md=""" ### Daily Sales Pipeline Processes previous day's sales. **Owner**: data-engineering **SLA**: complete by 08:00 UTC **Runbook**: confluence.company.com/airflow/daily-sales """, ) def daily_sales_pipeline(): @task() def extract_sales(ds=None) -> dict: rows = query_postgres( f"SELECT * FROM sales WHERE date='{ds}'") s3_path = write_to_s3(rows, ds) return {'row_count': len(rows), 's3_path': s3_path} @task() def validate_data(extract_result: dict, ds=None): assert extract_result['row_count'] > 0, \ f"Zero rows for {ds} — is upstream source healthy?" run_dbt = BashOperator( task_id='run_dbt', bash_command='dbt run --select tag:daily --vars \'{"date":"{{ ds }}"}\'', ) result = extract_sales() validated = validate_data(result) validated >> run_dbt daily_sales_pipeline()
ParameterRule
start_dateAlways static. datetime(2024,1,1) — never datetime.now()
catchupAlways False unless explicitly backfilling
max_active_runs1 for sequential pipelines — prevent data races
dagrun_timeoutAlways set — prevents zombie runs accumulating
doc_mdRunbook link required for critical DAGs
tagsteam + domain + frequency minimum
⚠ The catchup trap. Deploy with start_date=datetime(2023,1,1) and catchup=True (the default). Airflow immediately creates ~365 runs flooding every worker slot. Always set catchup=False explicitly. Always.
💡 Dependency patterns: a >> b sequential. a >> [b,c] fan-out. [a,b] >> c fan-in (c waits for both). Tasks inside a list run in parallel — they have no dependency on each other.
04 Idempotency & Determinism ★ SENIOR ESSENTIAL
The most important concept in production data engineering. This is the difference between a pipeline you trust and one you fear.
🔁 The idempotency test — apply it to every task you write

Run your task 10 times for the same data window. Does it produce the same final result as running it once? If yes: your task is idempotent. If no: you have a bug that will eventually cause a 3am page.

This matters because tasks fail and get retried. Backfills re-run historical data. Engineers manually clear and re-run tasks to fix issues. Every one of these scenarios runs your task more than once. If "run twice" produces different results than "run once," your data is wrong and you may not know it.

🏗 The contractor test: a good contractor can redo a tile they messed up. They remove the bad tile and re-lay it correctly. The end result is identical to if they'd done it right the first time. A bad contractor just lays the new tile on top — now you have two tiles, and the wall is wrong. Your tasks need to be the good contractor.
❌ Non-idempotent patterns — will cause incidents
# ❌ Plain INSERT — run twice = duplicate rows cursor.execute(""" INSERT INTO sales_summary (date, revenue) SELECT date, SUM(revenue) FROM sales WHERE date = '{{ ds }}' GROUP BY date """) # ❌ Blind append to a file — run twice = double data df.to_csv(f"s3://reports/{{ ds }}/summary.csv", mode='a') # ❌ Using datetime.now() for date logic def process(**context): today = datetime.now().strftime('%Y-%m-%d') # WRONG query_data(today) # backfill 30 days → every run queries today's data # ❌ Global counter with no reset def update_count(): current = get_counter() set_counter(current + 1) # run twice = counted twice
✅ Idempotent patterns — run these 10 times safely
# ✅ UPSERT — run twice = same result cursor.execute(""" INSERT INTO sales_summary (date, revenue) SELECT date, SUM(revenue) FROM sales WHERE date = '{{ ds }}' GROUP BY date ON CONFLICT (date) DO UPDATE SET revenue = EXCLUDED.revenue """) # ✅ Overwrite partition — run twice = same file df.to_parquet(f"s3://warehouse/sales/date={{ ds }}/data.parquet") # S3 overwrites by key — running again just replaces it # ✅ Always use {{ ds }}, never datetime.now() def process(ds=None, **context): query_data(ds) # correct regardless of when task runs # ✅ DELETE + INSERT pattern def load(ds=None): conn.execute( f"DELETE FROM sales_summary WHERE date = '{ds}'") conn.execute( f"INSERT INTO sales_summary ... WHERE date = '{ds}'") # running again: delete same rows, insert fresh = identical
The Idempotency Checklist — before merging any DAG
  • All date/time logic uses {{ ds }} or context['ds'], never datetime.now()
  • Database writes use UPSERT, MERGE, or DELETE+INSERT — never bare INSERT
  • File writes use date-partitioned paths (overwrite by key, not append)
  • Any aggregation uses the data window, not "all records" or "latest"
  • External API calls use the data window as the query parameter
  • Task can be manually cleared and re-run without corrupting data
  • Backfilling 30 days produces the same result as 30 individual runs
  • Running the task on an already-processed date doesn't change the output
  • If using S3, writing to the same key replaces — never appends to — the object
  • Test: run the task twice in your local compose. Compare outputs. Identical?

The Poison Pill pattern: a task fails. The bug is in the source data itself — that day's extract has bad schema that your transform can't handle. Airflow retries it. It fails again. Retries again. Fails. Every retry burns a worker slot and writes noise to your alerting system. Meanwhile all downstream tasks are blocked and your SLA is breaching.

The fix protocol: (1) set retries=0 on that task instance, (2) clear it to stop the retry loop, (3) fix the root cause in the source or transform logic, (4) clear the task to re-run with the fix. This is a workflow every senior runs at least monthly.

↯ The Three Sins of Airflow Development ★ SENIOR ESSENTIAL
These three patterns are responsible for the majority of production incidents. Know them. Enforce them in code review.
Sin #1 — Top-Level Code

A junior adding requests.get() at the top of a DAG file will DoS your own scheduler. Every DAG file is imported every 30 seconds. HTTP calls, DB connections, and expensive imports at module level exhaust connection pools and peg CPU — for the entire scheduler, affecting every DAG in your organisation.

# ❌ Kills the scheduler # runs every 30s per parse cycle config = requests.get( "http://config-api/settings" ).json() DB_CONN = psycopg2.connect( "postgresql://prod-db/..." ) with DAG('my_dag', ...) as dag: ...
# ✅ Correct: inside callable @task def fetch_config(): config = requests.get( "http://config-api/settings" ).json() return config with DAG('my_dag', ...) as dag: fetch_config()
⚠ The Rule: if it makes a network call, opens a file, or imports a heavy library — it lives inside a task callable. No exceptions. This is enforced in CI via the DAG integrity test suite.
Sin #2 — Variable.get() in a Loop

Every Variable.get() call is a synchronous database query. With 100 tasks each calling it once, you're hitting the metadata DB 100 times every 30-second parse cycle — 200 extra queries per minute from one DAG. At scale, this saturates your DB IOPS before any actual pipeline work runs.

# ❌ 100 DB hits per parse with DAG('pipeline', ...) as dag: for table in TABLES: PythonOperator( task_id=f'load_{table}', python_callable=lambda: load( Variable.get('env'), # ← DB hit table ) )
# ✅ Zero DB hits at parse time with DAG('pipeline', ...) as dag: for table in TABLES: BashOperator( task_id=f'load_{table}', # Jinja: resolved at runtime only bash_command=f'load.py ' '--env {{ var.value.env }}' f' --table {table}' )
💡 The Rule: use Jinja {{ var.value.key }} in operator templates, or fetch inside task callables. Never at module level or in operator constructor arguments.
Sin #3 — Database-as-a-Queue

XCom values are stored as BLOBs in the metadata database. Pushing a 1GB DataFrame crashes the DB. Even pushing a 50MB object will slow every metadata query across your entire Airflow installation — task state transitions, DAG parsing, UI rendering — everything that touches the DB degrades.

# ❌ Will crash the metadata DB @task def extract() -> pd.DataFrame: df = pd.read_parquet("s3://...") # returning a DataFrame = pushing # it to the DB as a BLOB return df # ← potential GB in DB @task def transform(df: pd.DataFrame): df['x'] = df['x'] * 2 # ← pulled from DB
# ✅ Pass the path. Never the data. @task def extract(ds=None) -> str: df = pd.read_parquet("s3://raw/") path = f"s3://staging/{ds}/data.parquet" df.to_parquet(path) return path # ← 50 bytes in DB ✓ @task def transform(path: str): df = pd.read_parquet(path) # ← from S3 df['x'] = df['x'] * 2
⚠ The Rule: XCom carries the address of the data, never the data itself. Row counts, S3 paths, status flags — fine. Anything that could exceed 1KB — write it to storage first.
05 Operators
Pre-built task templates. The vocabulary of Airflow.
PythonOperator — the most flexible

Calls any Python callable. Return value auto-pushed to XCom. Gets the full Airflow context — logical date, task instance, run ID, everything.

from airflow.operators.python import PythonOperator def process(ds=None, ds_nodash=None, **context): # ds = '2024-01-15' (YYYY-MM-DD) # ds_nodash = '20240115' (for S3 paths) # context['ti'] = task instance # context['dag_run'].conf = runtime config df = pd.read_parquet(f"s3://landing/{ds}/") df.to_parquet(f"s3://processed/{ds}/summary.parquet") return len(df) # → XCom auto-push op = PythonOperator( task_id='process', python_callable=process)
BashOperator — shell with Jinja

Runs shell in a subprocess. Non-zero exit fails the task. Last stdout line → XCom when do_xcom_push=True.

from airflow.operators.bash import BashOperator run_dbt = BashOperator( task_id='run_dbt', bash_command=""" cd /opt/dbt dbt run --select tag:daily \\ --vars '{"run_date": "{{ ds }}"}' """, env={'DBT_PROFILES_DIR': '/opt/dbt'}, ) load = BashOperator( task_id='load', bash_command='aws s3 cp /tmp/{{ ds }}.parquet s3://dw/raw/{{ ds }}/', )
SQLExecuteQueryOperator — SQL on any DB
from airflow.providers.common.sql.operators.sql import SQLExecuteQueryOperator # Run a SQL file — keep DAG code clean load_staging = SQLExecuteQueryOperator( task_id='load_staging', conn_id='postgres_dwh', sql='sql/load_sales.sql', parameters={'run_date': '{{ ds }}'}, ) # Data quality check: empty result = failure check = SQLExecuteQueryOperator( task_id='row_count_check', conn_id='postgres_dwh', sql=""" SELECT 1 FROM sales WHERE date='{{ ds }}' HAVING COUNT(*) > 0; """, )
KubernetesPodOperator — full isolation

One fresh container per task run. Any image, any Python version, any dependencies — isolated from Airflow itself.

from airflow.providers.cncf.kubernetes.operators.pod import KubernetesPodOperator from kubernetes.client import models as k8s train = KubernetesPodOperator( task_id='train_model', name='training-pod', namespace='data-engineering', image='myregistry/ml-trainer:2.1.0', cmds=['python', '/app/train.py'], arguments=['--date', '{{ ds }}'], container_resources=k8s.V1ResourceRequirements( requests={'memory': '8Gi', 'cpu': '4'}, limits={'memory': '16Gi'}, ), get_logs=True, is_delete_operator_pod=True, in_cluster=True, )

06 TaskFlow API
Airflow 2.0+. Write DAGs that read like normal Python. Half the boilerplate.
❌ Classic style — too much ceremony
def extract(**context): rows = fetch(context['ds']) context['ti'].xcom_push(key='rows', value=rows) def transform(**context): rows = context['ti'].xcom_pull( task_ids='extract', key='rows') context['ti'].xcom_push(key='clean', value=clean(rows)) with DAG('pipeline', ...) as dag: t1 = PythonOperator(task_id='extract', python_callable=extract) t2 = PythonOperator(task_id='transform', python_callable=transform) t1 >> t2 # every dependency is manual
✅ TaskFlow — readable, automatic XCom wiring
@dag(schedule='@daily', start_date=datetime(2024,1,1), catchup=False) def pipeline(): @task def extract(ds=None) -> list: return fetch(ds) # return = auto XCom push @task def transform(rows: list) -> list: return [clean(r) for r in rows] @task def load(cleaned: list) -> int: write_to_dw(cleaned) return len(cleaned) # function calls = dependencies + XCom wiring in one step load(transform(extract())) pipeline()
Dynamic Task Mapping — the game changer for variable-scale workloads

You have 1,000 files in S3. You don't write 1,000 tasks. You write one task and let Airflow create one instance per file at runtime using .expand(). Each instance gets its own state, logs, and retry count. One region fails? Only that instance retries.

⚠ XCom limit applies to the expand list. The list passed to .expand() must fit in XCom (~1MB default). Pass paths as strings — never file contents. For 10,000+ items, write the list to S3 and pass the path.
💡 The fan-in aggregate(results) task automatically receives a list of all mapped instance return values. Airflow handles the collection — you just receive the list.
🎯 The Unpredictable Volume Scenario: "We have a bucket. Sometimes 5 files arrive, sometimes 5,000." Don't write a loop that creates 5,000 operators at DAG-write time. Use .expand() — the task count is determined at runtime from real data, with zero code changes or redeployments. This is how you handle variable-scale workloads correctly.
@dag(schedule='@daily', start_date=datetime(2024,1,1), catchup=False) def parallel_processor(): @task def get_files(ds=None) -> list[str]: # Returns a list — one instance created per item return list_s3_files(f"s3://landing/{ds}/") @task def process_file(s3_path: str) -> dict: # Runs once per path, in parallel df = pd.read_parquet(s3_path) return {'path': s3_path, 'rows': len(df)} @task def aggregate(results: list[dict]) -> None: # All mapped results collected here automatically total = sum(r['rows'] for r in results) log.info(f"Processed {total} total rows") files = get_files() results = process_file.expand(s3_path=files) aggregate(results) parallel_processor()
07 Scheduling
Cron, timedeltas, Dataset triggers, and backfills.
All schedule types
# Preset strings schedule='@once' # run exactly once schedule='@hourly' # 0 * * * * schedule='@daily' # 0 0 * * * (midnight UTC) schedule='@weekly' # 0 0 * * 0 schedule=None # manual trigger only # Custom cron (minute hour day month weekday) schedule='0 6 * * *' # 6AM UTC daily schedule='0 6 * * 1-5' # 6AM Mon-Fri only schedule='0 */4 * * *' # every 4 hours schedule='*/15 * * * *' # every 15 minutes # timedelta (Airflow 2.4+) schedule=timedelta(hours=6) # Dataset trigger (Airflow 2.4+) — preferred over ExternalTaskSensor from airflow.datasets import Dataset raw = Dataset("s3://dw/raw/sales/") schedule=raw # runs when this dataset is updated schedule=[ds_a, ds_b] # runs when BOTH are updated
Data-Aware Scheduling — trigger on data, not time

Upstream DAG marks a task with outlets=[dataset]. Downstream DAG sets schedule=[dataset]. Airflow fires downstream automatically when the outlet is updated — no polling, no ExternalTaskSensor, no hardcoded time offsets.

# Upstream declares what it produces @task(outlets=[Dataset("s3://dw/raw/sales/")]) def ingest(ds=None): write_to_s3(ds) # Downstream triggered automatically @dag(schedule=[Dataset("s3://dw/raw/sales/")], ...) def transform(): ...
Backfill
airflow dags backfill \ --dag-id daily_sales \ --start-date 2024-01-01 \ --end-date 2024-01-31 # Test single task without saving state airflow tasks test daily_sales extract_sales 2024-01-15
⚠ Backfills only work if tasks are idempotent — see Section 4. Running a plain-INSERT task for 30 historical days produces 30x duplicate rows.
08 Executors
The most important infrastructure decision. Choose for scale, not familiarity.
⚡ LocalExecutor

Parallel tasks on one machine. Subprocess per task. Requires PostgreSQL. No extra infra.

✓Good for: small teams, <50 concurrent tasks, getting started fast.
🌐 CeleryExecutor

Distributed worker fleet. Redis/RabbitMQ broker. Scale horizontally by adding workers.

✓The standard production choice. Most widely deployed.
☸️ KubernetesExecutor

One pod per task. Full container isolation. Pay only for what runs. Deleted after completion.

✓Best for mixed environments, strict isolation, cloud-native.

10 XCom — Passing Data Between Tasks
For paths and counts. Never for DataFrames. This distinction ships incidents.
Push and pull — both styles
# TaskFlow: return = push, argument = pull (preferred) @task def extract(ds=None) -> dict: rows = fetch(ds) path = write_s3(rows, ds) return {'s3_path': path, 'row_count': len(rows)} @task def transform(result: dict) -> str: df = pd.read_parquet(result['s3_path']) # pull path, not data return write_s3(df, 'silver') # Classic: explicit push/pull def extract_classic(**context): context['ti'].xcom_push(key='path', value=write_s3(...)) def transform_classic(**context): path = context['ti'].xcom_pull( task_ids='extract_classic', key='path')
What to push — and what never to push
DataSizeSafe?
Row count, status flagBytes✓ Always fine
S3/GCS file path~100 bytes✓ Always fine

The correct pattern for any large data: Task A writes to S3 → pushes the path (string) to XCom → Task B pulls the path → reads from S3. Scales to any data size. XCom stays tiny.
11 Branching & Flow Control
If/else in pipelines. Trigger rules. The branch-join trap.
@task.branch — conditional routing
@dag(schedule='@daily', start_date=datetime(2024,1,1), catchup=False) def conditional_pipeline(): @task.branch def route(ds=None) -> str: day = datetime.strptime(ds, '%Y-%m-%d').weekday() return 'full_run' if day == 0 else 'incremental_run' @task def full_run(): run_full_model() @task def incremental_run(): run_incr_model() # CRITICAL: use this trigger rule to rejoin branches # Without it, 'finish' is skipped because one upstream is skipped @task(trigger_rule='none_failed_min_one_success') def finish(): slack_notify("Training complete") r = route() a, b = full_run(), incremental_run() r >> [a, b] [a, b] >> finish() # trigger_rule on finish ↑ conditional_pipeline()

12 Sensors
Always reschedule mode. Always set timeout. No exceptions.
👁 The one rule that matters

mode='poke' (default) holds a worker slot the entire wait time. 20 sensors waiting 4 hours = 20 worker slots consumed for zero work. Real tasks queue up. The system looks "busy" while doing nothing. Always use mode='reschedule' — it checks, releases the slot, gets re-queued after the interval. Zero slot waste.

S3KeySensor

Day-to-Day Workflow — the three-environment contract

LOCAL

  • Run pytest tests/test_dag_integrity.py before every commit — catches import errors and policy violations instantly
  • Use airflow dags test my_dag 2024-01-15 to execute a full DAG run locally without needing the scheduler or metadata DB
  • Verify idempotency: run the same task twice for the same date, compare output. If different — you have a bug.
  • Test sensor logic with airflow tasks test — confirm poke returns True for valid data and False otherwise

CI (pull request gate)

  • DAG integrity suite passes: no import errors, no cycles, catchup=False, dagrun_timeout set
  • No datetime.now() in any DAG file — caught by static analysis or the start_date freshness test
  • No Variable.get() at module level — enforced via AST lint rule
  • Reviewer explicitly checks: are all tasks idempotent? Is any XCom push potentially large?

PRODUCTION

  • Never backfill more than 5 runs via the UI. It locks the UI for everyone and creates a thundering herd on workers. Use the CLI for anything larger.
  • CLI backfill with --max-active-runs 4 — controlled parallelism, doesn't saturate workers
  • Monitor airflow dags list-import-errors as a health check in your deployment pipeline
  • If a SQLite connection string or SequentialExecutor appears in any PR targeting production — reject immediately. No exceptions.
13 Zombies & Poison Pills ★ SENIOR ESSENTIAL
Every experienced Airflow engineer has fought both. Know them before your first 3am page.
🧟 Zombie Tasks
The task the DB thinks is running. The worker process is dead.

A Zombie is a task instance stuck in running state while the process that was actually executing it is gone. The DB was never updated because the worker was killed externally — OOM, pod eviction, spot instance termination, node failure — before it could write a terminal state.

How to spot zombies in logs:
# In scheduler logs — the dead giveaway: Detected zombie job: ... TaskInstance: my_dag.my_task [running] # Or in worker logs: Executor reports task instance my_dag.my_task finished with state 'failed' but the task is not in a terminal state # CLI: find all zombies now airflow tasks list-running --dag-id my_dag | \ grep -E "running.*[0-9]{2}min" # stuck long
Resolution protocol
# Option 1: Let the zombie detector handle it # airflow.cfg: [scheduler] zombie_detection_interval = 10.0 # Default: checks every 10s, marks stuck tasks as 'failed' # after zombie_task_threshold_secs (default 300s) # Option 2: Manual clear if you need it resolved now airflow tasks clear my_dag \ --task-id my_task \ --start-date 2024-01-15 \ --end-date 2024-01-15 # Option 3: Mark failed directly via CLI airflow tasks set-state my_dag my_task failed \ --run-id scheduled__2024-01-15T06:00:00 # Then investigate root cause: # - Check worker system memory (dmesg, CloudWatch, Datadog) # - Check K8s events: kubectl get events -n airflow # - Check spot/preemptible instance termination logs
⚠ Zombies from OOM are silent. The task disappears with no Python exception, no stack trace, no Airflow log output. If a task vanishes with no error in the Airflow UI, your first move is to check worker system memory metrics — not Airflow logs. The OOM killer leaves no Python trace.
☠️ Poison Pill Tasks
The task that will always fail for this data. Burning slots on every retry.

A Poison Pill is a task where the failure is in the data, not the code. The source file has bad schema. The API returned garbage. The expected partition doesn't exist. Airflow will faithfully retry it retries times — each retry burns a worker slot, writes noise to your alert channel, and delays everything downstream.

# Recognition: rapid fail-retry-fail-retry pattern # in the UI with no change between runs # ── Emergency stop protocol ──────────────── # Step 1: Stop the retry loop immediately # In UI: task → Edit → retries = 0 # Or via CLI: airflow tasks set-state my_dag poison_task failed \ --run-id scheduled__2024-01-15T06:00:00 # Step 2: Diagnose the root cause # - Check source data integrity # - Check upstream system health # - Check if schema changed # Step 3: Fix root cause in code or source data # Step 4: Clear and re-run with fix airflow tasks clear my_dag \ --task-id poison_task \ --start-date 2024-01-15 --end-date 2024-01-15
Zombie vs Poison Pill — quick distinction
ZombiePoison Pill
DB staterunningfailed repeatedly
Worker processDead (killed externally)Runs, fails cleanly
Log outputTruncated or absentFull Python traceback
Root causeOOM, eviction, node failureBad data, changed schema, API error
FixIncrease memory, handle preemptionFix data source or transform logic

Prevention over cure: Set dagrun_timeout on every DAG. This is your backstop — a zombie or poison pill can't hold the pipeline hostage past the timeout. When it trips: the run fails, you get an alert, you investigate. Without it, a zombie in "running" state can block your DAG for days.

The "Task Lag" diagnosis path

Open the Gantt chart for a recent DAG run. The colored bars tell the whole story:

Long yellow bar (queued) — workers are full or broker is slow. Add workers or increase parallelism.
Long blue bar (running) — the task itself is slow. Profile the code, check for N+1 queries, consider streaming instead of full loads.
Instant red (failed) — import error or immediate exception. Check logs — if there are none, it's an OOM kill. Check worker system metrics.
Long grey gap before queued — scheduler parse latency. Your DAG files are too heavy. Check for module-level I/O.
14 CI/CD & DAG Testing ★ SENIOR ESSENTIAL
If airflow dags list-import-errors shows anything in production, the engineer who pushed it is buying coffee.
🧪 DAG integrity tests — the minimum viable CI pipeline

We don't drop files in a folder and hope. Every DAG goes through CI before touching the scheduler. The core test suite catches 80% of production incidents: import errors, circular dependencies, catchup=True accidents, missing timeouts, sensors in poke mode. These run in under 60 seconds and block the merge if anything fails.

Core DAG integrity test suite — paste this into your CI
# tests/test_dag_integrity.py import pytest from airflow.models import DagBag from datetime import datetime, timedelta @pytest.fixture(scope='session') def dagbag(): return DagBag(include_examples=False) def test_no_import_errors(dagbag): """Any DAG with an import error would brick the scheduler.""" assert dagbag.import_errors == {}, \ f"Import errors: {dagbag.import_errors}" def test_dag_count(dagbag): assert len(dagbag.dags) > 0, "No DAGs found in the bag" def test_all_dags_have_catchup_false(dagbag): """Prevent the "500 backfill runs" incident.""" offenders = [ dag_id for dag_id, dag in dagbag.dags.items() if dag.catchup is not False ] assert offenders == [], \ f"DAGs with catchup=True: {offenders}" def test_all_dags_have_timeout(dagbag): """A DAG without timeout can hold the scheduler hostage.""" offenders = [ dag_id for dag_id, dag in dagbag.dags.items() if dag.dagrun_timeout is None ] assert offenders == [], \ f"DAGs missing dagrun_timeout: {offenders}" def test_all_dags_have_static_start_date(dagbag): """start_date=datetime.now() causes a new run every scheduler cycle.""" now = datetime.utcnow() for dag_id, dag in dagbag.dags.items(): delta = (now - dag.start_date.replace(tzinfo=None)).total_seconds() assert delta > 60, \ f"{dag_id} has a dynamic start_date (set within the last 60s)" def test_all_dags_have_owner(dagbag): offenders = [ dag_id for dag_id, dag in dagbag.dags.items() if dag.owner in (None, '', 'airflow') ] assert offenders == [], \ f"DAGs with default/missing owner: {offenders}" def test_no_cycles(dagbag): """Circular dependency crashes the scheduler.""" for dag_id, dag in dagbag.dags.items(): dag.test_cycle() # raises if cycle detected

Production checklist — gate every DAG deployment on this

DAG Code

  • Static start_date — never datetime.now()
  • catchup=False explicit — no accidents
  • max_active_runs=1 for sequential pipelines
  • dagrun_timeout set — no zombie pipeline runs
  • All I/O inside callables — nothing at module level
  • Use {{ ds }}, never datetime.now() in task logic
  • Tasks are idempotent — proven via "run twice" test
  • retries and retry_delay set for transient failures
  • doc_md with owner + runbook link
  • CI test suite passes: no import errors, no cycles

Infrastructure

  • PostgreSQL metadata DB — never SQLite
  • AIRFLOW__CORE__FERNET_KEY set — credentials encrypted at rest
  • Credentials in secrets backend — not the metadata DB
  • CeleryExecutor or KubernetesExecutor — never LocalExecutor at scale
  • Remote logging to S3/GCS — workers are ephemeral
  • Scheduler heartbeat monitored with alerting
  • on_failure_callback set for all critical DAGs
  • SLAs defined on time-sensitive pipelines
  • All sensors use mode='reschedule'
  • Worker memory provisioned with headroom for peak tasks
