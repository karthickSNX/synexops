Chapter One

What is Apache Kafka?

📖 Why "Kafka"? Named after the author Franz Kafka — and that's a clue about what it does

"I am a cage, in search of a bird." — Franz Kafka

Jay Kreps, one of Kafka's creators at LinkedIn, named it after the author Franz Kafka because the system is "optimized for writing." Kafka the author was prolific at writing, and Kafka the software is designed for fast, high-throughput, durable writes. The name stuck. Today it powers the data infrastructure of Uber, Netflix, LinkedIn, Airbnb, and thousands of other companies — all because of one brilliant insight about how to move data reliably at scale.

💡 The Practical Explanation — What Kafka Actually Solves

Say you're building a food delivery app. When a user places an order, about six different things need to happen: the restaurant gets notified, the payment service charges the card, the driver-assignment system finds a nearby driver, the analytics system logs the order for reports, the fraud-detection system checks for suspicious patterns, and a push notification goes to the user's phone.

The naive approach: your order service calls all six systems directly, one by one, in a chain. Now you have a problem. If the notification service is slow, your order service waits. If the analytics service crashes, your order fails. You've tightly coupled six different systems into one fragile chain — and every new system you add means changing the order service code again.

Kafka's approach: your order service writes one event — "order #4521 placed by user alice, $47.99" — to Kafka and immediately returns. Done. Each of the six downstream systems independently reads that event from Kafka at its own pace. The order service doesn't know or care who reads it. You can add a seventh system later without touching the order service at all.

That's the core idea: Kafka is a durable, centralized event log that lets services communicate without being directly connected to each other. Events are written once and can be read by any number of consumers, independently, even days later. This is what "decoupling" means in practice — and it's why systems at scale can't function without something like it.

🆚 Traditional Queue vs Kafka — What's the Difference?

Most people have heard of message queues (like RabbitMQ or SQS). Kafka looks similar from the outside but works fundamentally differently. This difference is the whole point.

❌ Traditional Queue (e.g. RabbitMQ)
Post office model

A letter arrives → one worker picks it up → the letter is gone forever. Nobody else can read it. If the worker drops it, it's lost. You can't replay yesterday's letters.

vs
✅ Kafka (Distributed Log)
Newspaper archive model

Every message is written to a permanent log. 10 different consumers can each read it independently. The message stays for days. You can replay last week's data any time. Reading never deletes anything.

💡 The key question: does your system need one consumer to process each message (queue) — or do multiple independent systems each need their own copy of every event (log)? If the latter, you want Kafka.

The 4 Core Building Blocks — Events, Topics, Partitions, Brokers
⚡ Events — The Thing Kafka Actually Stores

An event (also called a message or record) is the fundamental unit in Kafka. It's a piece of data representing something that happened — a fact, an action, a state change. Events are immutable: once written, never changed.

Every Kafka event has three parts:

  • K
    Key — optional identifier, usually a user ID or order ID. Events with the same key always land in the same partition, guaranteeing they stay in order relative to each other.
  • V
    Value — the actual payload. Could be JSON, Avro, Protobuf, plain text — anything serialized to bytes. This is where your data lives.
  • T
    Timestamp — when the event was created or ingested. Used for time-windowed aggregations and log compaction.
What an event looks like
// A single Kafka event
{
  key: "user-alice",
  value: {
    "event": "order_placed",
    "order_id": 4521,
    "amount": 47.99
  },
  timestamp: "2024-03-15T14:22:01Z",
  offset: 1847  // position in partition
}
✅ Events are immutable. Once written, they cannot be edited. This is what makes Kafka reliable — the log is the truth.
📁 Topics — The Named Channels That Organise Events

A topic is a named, durable stream of related events. Think of it like a table in a database, or a folder on your filesystem — but for a continuous stream of time-ordered facts. Producers write to a topic; consumers subscribe to read from it.

In a real system you'd have many topics, each for a different kind of event:

orders — every order placed by any user

driver-locations — GPS pings from delivery drivers
notifications — push / email / SMS to be sent

A single topic can receive millions of events per second — far more than one server could handle sequentially. Kafka solves this by splitting each topic into partitions: independent sub-logs that can live on different servers and be read by different consumers in parallel.

✅ Same key → same partition, always. All of alice's orders land in P0 in arrival order. This guarantees that Consumer A sees alice's orders in sequence — even though the topic has 3 partitions running in parallel.
⚠️ Order is only guaranteed within one partition. If alice and bob both place orders at the same time, you can't know which partition is ahead. Design your system so that per-user or per-entity ordering (within a partition) is what you actually need — not global ordering across the whole topic.
🖥 Brokers — The Servers That Store and Serve Everything

A broker is just a Kafka server — a process running on a machine that stores partition data and handles producer/consumer requests. A Kafka cluster is typically 3 or more brokers working together.

Each partition has one leader broker (handles all reads and writes for that partition) and N-1 follower brokers (passively replicate data for backup). If the leader dies, a follower is automatically promoted.

  • Partitions are spread across brokers for even load
  • Each broker leads some partitions, follows others
  • Adding a broker = more storage + more throughput capacity
  • Kafka 3+ uses KRaft (no ZooKeeper needed) for cluster coordination
🏆 Production minimum: 3 brokers, RF=3, min.insync.replicas=2. With this setup, you can lose one broker with zero data loss and zero downtime.

🔁 Replication & Fault Tolerance — How Kafka Survives Broker Failures
  • RF
    replication.factor — number of broker copies per partition. RF=3 means 3 brokers each hold the partition. Survive 2 broker losses without data loss.
  • ISR
    In-Sync Replicas (ISR) — the set of replicas that are caught up to the leader. Only ISR members are eligible to become the new leader on failover.
  • L
    Leader — the broker currently handling reads and writes for a partition. If it dies, Kafka's controller elects a new leader from the ISR within seconds.
  • ✏
    min.insync.replicas=2 — a producer with acks=all won't consider a write successful until 2 replicas have it. Prevents silent data loss on leader failure.
🏆 Production baseline: RF=3, min.insync.replicas=2, acks=all. This gives you durability guarantees while tolerating 1 broker failure during writes.
# Production broker/topic settings default.replication.factor=3 min.insync.replicas=2 unclean.leader.election.enable=false # → only ISR members can become leader # → prevents data loss from stale replicas

Producers — Writing to Kafka

Before reading the code
bootstrap.serversThe initial broker list your client connects to just to discover the full cluster topology. You don't need all brokers listed — just 2–3 for resilience.
SerializationKafka messages are raw bytes. You convert your Python object → bytes before producing (json.dumps(obj).encode()), and bytes → object after consuming.
Delivery callbackA function Kafka calls asynchronously after a message is confirmed (or failed). Your code keeps running; the callback fires when the broker responds.
flush() vs poll()flush() blocks until all buffered messages are delivered. poll(0) triggers the event loop (runs callbacks) without blocking — call it in long loops.
🏭 Producer — Python (confluent-kafka)
# pip install confluent-kafka from confluent_kafka import Producer import json conf = { 'bootstrap.servers': 'localhost:9092', 'acks': 'all', # wait for all ISR replicas 'retries': 5, # retry on transient errors 'linger.ms': 10, # batch for 10ms before sending 'compression.type': 'lz4', # compress batches 'batch.size': 65536, # 64KB batch max 'enable.idempotence': True, # dedup retries on broker } p = Producer(conf) def on_delivery(err, msg): if err: print(f'❌ Delivery failed: {err}') else: print(f'✅ {msg.topic()}[{msg.partition()}] @ offset {msg.offset()}') # Produce a message order = {"id": 123, "amount": 49.99, "user": "alice"} p.produce( topic='orders', key='user-alice', # same key → same partition → ordering value=json.dumps(order).encode('utf-8'), callback=on_delivery ) p.poll(0) # trigger delivery callbacks without blocking p.flush() # block until all messages are delivered

💡 Key routing: if you provide a key, Kafka hashes it to a partition. Same key → same partition → ordered delivery for that key. No key → round-robin across partitions.
⚠️ Pitfall: Producing in a tight loop without poll()

The confluent-kafka producer has an internal queue. If you call produce() in a tight loop without ever calling poll(), the queue fills up and produce() blocks. Call p.poll(0) periodically inside long loops to drain callbacks and keep the queue healthy.

for record in large_dataset: p.produce('topic', value=json.dumps(record).encode()) p.poll(0) # ← don't skip this in loops p.flush() # wait for all at the end

Chapter Four

Consumers — Reading from Kafka

How committing offsets works — critical for beginners

After your consumer reads a message and processes it, it needs to commit the offset — telling Kafka "I've successfully handled up to message #47 in partition 0." This is how Kafka knows where to resume after a consumer crash or restart.

Auto-commit is dangerous in most real applications. With auto-commit enabled, Kafka commits the offset every N milliseconds — even if your processing code hasn't finished or errored. If your code crashes between the auto-commit and actually finishing, that message is silently skipped forever. Use manual commit in production.

📱 Consumer — Python (at-least-once, manual commit)
from confluent_kafka import Consumer, KafkaError import json conf = { 'bootstrap.servers': 'localhost:9092', 'group.id': 'order-processor', # consumer group name 'auto.offset.reset': 'earliest', # start from oldest if no commit 'enable.auto.commit': False, # manual commit = safe 'session.timeout.ms': 30000, 'max.poll.records': 500 } c = Consumer(conf) c.subscribe(['orders']) try: while True: msg = c.poll(timeout=1.0) # wait up to 1s for a message if msg is None: continue # poll timeout, no message if msg.error(): # PARTITION_EOF is normal — not an error if msg.error().code() == KafkaError._PARTITION_EOF: continue raise Exception(msg.error()) # ← process here, THEN commit order = json.loads(msg.value().decode('utf-8')) print(f"Order #{order['id']} — ${order['amount']}") save_to_db(order) # commit AFTER successful processing c.commit(asynchronous=False) finally: c.close() # always close — triggers graceful rebalance

📍 Offset Management — Seeking, Resetting, Replaying

Consumers don't have to read sequentially from where they left off. You can programmatically seek to any offset — useful for replays, debugging, or recovering from bad deploys.

💡 Committed offsets are stored in a special Kafka topic called __consumer_offsets. This is why Kafka (not your app) tracks where each group is — it's durable across restarts.
# Seek to beginning of specific partition from confluent_kafka import TopicPartition c.assign([TopicPartition('orders', 0)]) c.seek(TopicPartition('orders', 0, 0)) # offset 0 # Seek to end (skip all existing messages) c.seek_to_end(TopicPartition('orders', 0)) # CLI: reset entire group offset # kafka-consumer-groups.sh \ # --reset-offsets --to-earliest \ # --group my-group --topic orders \ # --execute --bootstrap-server localhost:9092

Chapter Five

Delivery Semantics

Why this matters — networks lie

Networks fail. Brokers restart. Producers retry. When a failure happens mid-write, what guarantee does your consumer get? The three semantics below are a fundamental trade-off between performance and correctness. Choose based on what data loss or duplication would actually mean for your use case — there's no free lunch.

Idempotent consumerOne that produces the same outcome even if it processes the same message twice. Example: an upsert on a DB primary key. Running it twice is safe — the second write just overwrites with the same value.
Idempotent producerKafka assigns each producer a PID (producer ID) and tracks sequence numbers per partition. If a retry sends a duplicate, the broker silently deduplicates it. Enable with enable.idempotence=true.
🔁
At-Most-Once

Messages may be lost, never duplicated. Offset committed before processing. If your code crashes after commit, that message is gone.

  • Fastest — no retry overhead
  • Acceptable for metrics, analytics logs
'enable.auto.commit': True 'auto.commit.interval.ms': 5000 # offset committed on schedule # regardless of processing status
✅
At-Least-Once Most Common

Messages never lost, but may be duplicated on retry. Commit after successful processing. Design your consumer to be idempotent.

  • Reliable, practical, widely used
  • Deduplicate downstream by message ID
'enable.auto.commit': False # process message fully, then: c.commit(asynchronous=False) # if crash before commit → re-process # make your handler idempotent!
💎
Exactly-Once

Processed exactly once, guaranteed. Requires idempotent producers + Kafka transactions. Complex; adds latency. Use for financial data.

  • Financial transactions, billing
  • Any domain where duplicates = real harm
'enable.idempotence': True, 'transactional.id': 'my-txn-001' p.init_transactions() p.begin_transaction() p.produce('topic', value=b'data') p.commit_transaction()
💀 Dead Letter Queue (DLQ) — Handling Poison Messages

A poison message is one your consumer can't process — malformed JSON, unexpected schema, business rule violation. If you crash or skip it, you get stuck or lose data.

The DLQ pattern: on processing failure, write the bad message to a separate orders-dlq topic along with the error and metadata. Your main consumer keeps moving. A separate DLQ consumer handles retries or alerts humans.

  • Main pipeline never stalls on bad data
  • Failed messages preserved for investigation
  • Retry or discard from DLQ separately
def process_safe(msg, producer): try: data = json.loads(msg.value()) handle(data) c.commit(asynchronous=False) except Exception as e: # send to dead-letter topic dlq_payload = { "original": msg.value().decode(), "error": str(e), "topic": msg.topic(), "partition": msg.partition(), "offset": msg.offset(), } producer.produce( topic='orders-dlq', value=json.dumps(dlq_payload).encode() ) producer.flush() c.commit(asynchronous=False) # still commit!

Chapter Six

Schema Registry & Avro

Why schemas matter in Kafka

Without schema enforcement, producers and consumers are just exchanging raw bytes and hoping the format matches. In practice, a developer changes a field name on the producer side and the consumer silently starts returning None everywhere — or crashes. The Schema Registry is a central server where producers register their message schema (Avro, Protobuf, or JSON Schema). Consumers automatically fetch and validate the schema before deserializing. It's the contract enforcement layer between services.

AvroA compact binary serialization format with a JSON-defined schema. Much smaller than JSON on the wire. Requires the schema to deserialize — enforces contracts at the format level.
Schema evolutionChanging a schema over time without breaking existing producers or consumers. Avro supports adding optional fields (backward compatible) and removing optional fields (forward compatible).

💻 Avro Producer with Schema Registry
from confluent_kafka.avro import AvroProducer from confluent_kafka import avro value_schema_str = """ { "type": "record", "name": "Order", "fields": [ {"name": "id", "type": "int"}, {"name": "amount", "type": "double"}, {"name": "user", "type": "string"}, {"name": "status", "type": "string", "default": "pending"} ] }""" producer = AvroProducer({ 'bootstrap.servers': 'localhost:9092', 'schema.registry.url': 'http://localhost:8081' }, default_value_schema=avro.loads(value_schema_str)) producer.produce( topic='orders', value={ 'id': 1, 'amount': 49.99, 'user': 'alice', 'status': 'confirmed' } ) producer.flush()
✅ Schema evolution rules: adding a field with a default value is backward compatible (old consumers still work). Removing a required field or changing a type is a breaking change — the registry rejects it.

Chapter Seven

Kafka Streams & ksqlDB

Stream processing without a separate cluster

So far you've written producers and consumers by hand. But what if you need to filter, enrich, aggregate, or join event streams continuously? You have two Kafka-native options that don't require running a separate Spark or Flink cluster.

Kafka StreamsA Java/Scala library you embed in your application. No cluster to manage. Scales horizontally by adding instances. State stored in local RocksDB + backed up to a Kafka compacted topic.
ksqlDBA SQL engine that runs on Kafka. Write CREATE STREAM ... AS SELECT and Kafka continuously executes it. Great for dashboards, real-time aggregations, and simple enrichments without writing code.
KStreamAn unbounded stream of independent events (like an append-only log). Each record matters individually — think order events, clicks, logs. Join, filter, aggregate across time windows.
KTableA compacted changelog — only the latest record per key is kept. Think "current state of user X" not "history of all user X actions." Backed by a Kafka compacted topic.
🌊 Kafka Streams (Java) — Filter → Aggregate
// Java / Kafka Streams 3.x StreamsBuilder builder = new StreamsBuilder(); KStream<String, String> orders = builder.stream("orders-raw"); // 1. Filter: only high-value orders KStream<String, Order> highValue = orders .mapValues(v -> parseOrder(v)) .filter((k, order) -> order.amount > 100.0); // 2. Write enriched stream to new topic highValue.to("orders-high-value"); // 3. Count orders per user per 5-min window KTable<Windowed<String>, Long> counts = orders .groupBy((k, v) -> extractUser(v)) .windowedBy(TimeWindows.ofSizeWithNoGrace( Duration.ofMinutes(5))) .count(); KafkaStreams app = new KafkaStreams(builder.build(), props); app.start();
🗃 ksqlDB — SQL on Kafka Streams
-- 1. Create a stream from a Kafka topic CREATE STREAM orders ( id INT, amount DOUBLE, user VARCHAR ) WITH ( KAFKA_TOPIC='orders-raw', VALUE_FORMAT='JSON' ); -- 2. Continuous aggregation into a table CREATE TABLE hourly_spend AS SELECT user, SUM(amount) AS total, COUNT(*) AS num_orders FROM orders WINDOW TUMBLING (SIZE 1 HOUR) GROUP BY user EMIT CHANGES; -- 3. Push query (live stream) SELECT * FROM hourly_spend EMIT CHANGES; -- 4. Pull query (point-in-time snapshot) SELECT * FROM hourly_spend WHERE user='alice';
💡 Push vs Pull: EMIT CHANGES = a live subscription that keeps streaming new results. Without it = a one-time snapshot of current state.

Security

Three layers of Kafka security

Kafka security is built from three orthogonal concerns: encryption (nobody can read messages in transit), authentication (who are you?), and authorization (what are you allowed to do?). You typically need all three in production. They're configured independently.

🔒 Encryption — SSL/TLS

Encrypts data in transit between clients and brokers. Standard TLS — same as HTTPS.

# Producer config 'security.protocol': 'SSL', 'ssl.ca.location': '/certs/ca.pem', 'ssl.certificate.location': '/certs/client.pem', 'ssl.key.location': '/certs/client.key'
💡 Use SASL_SSL when you want both auth AND encryption (most common production setup).
🔑 Authentication — SASL

Verifies client identity. Three main mechanisms:

  • P
    PLAIN — username/password in config. Simple but transmits credentials in plaintext — always combine with SSL.
  • S
    SCRAM-SHA-256/512 — hashed credential exchange. Safer than PLAIN. Credentials stored in ZooKeeper/KRaft.
  • K
    Kerberos (GSSAPI) — enterprise SSO. Complex setup but integrates with existing Active Directory / LDAP infrastructure.
  • O
    OAUTHBEARER — token-based auth. Integrates with OAuth2 / OIDC providers like Keycloak or Okta.
🛡 Authorization — ACLs

Controls what each authenticated principal can do. Principle of least privilege — give each service only what it needs.

# Grant service-A read access to "orders" kafka-acls.sh \ --bootstrap-server localhost:9092 \ --add \ --allow-principal User:service-a \ --operation Read \ --topic orders # Grant write access to producer kafka-acls.sh \ --bootstrap-server localhost:9092 \ --add \ --allow-principal User:order-service \ --operation Write \ --topic orders # List all ACLs kafka-acls.sh \ --bootstrap-server localhost:9092 --list

Monitoring & Observability

The most important Kafka health metric

Consumer lag is the gap between the latest offset written to a partition and the offset a consumer group has committed. A lag of 0 means your consumer is caught up. A growing lag means your consumer is falling behind faster than it can process. This is your primary alerting signal for Kafka health — not CPU, not disk, not broker memory. Consumer lag.

⚠️ Common Production Problems & Fixes
ProblemSymptomFix
Lag keeps growing Consumer can't keep up with producer rate Scale consumer instances, increase partitions (then consumers), or optimize processing logic
Under-replicated partitions Under-replicated partitions > 0 Check broker health, network, disk I/O. Replace or restart sick broker.
Frequent rebalancing Consumers repeatedly dropping/rejoining Increase session.timeout.ms, reduce max.poll.records so processing finishes before heartbeat expires
Disk full on broker Broker crashes, partition leaders fail over Reduce log.retention.ms, add disk, or add brokers + rebalance leaders
Duplicate messages Consumer processes same event twice Make consumers idempotent (upsert by ID). Enable idempotent producer if duplicates are from retries.
