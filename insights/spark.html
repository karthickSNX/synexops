1 What is Apache Spark?
Start here — the mental model that makes everything else click
🔥 The Plain-English Version

Apache Spark is a distributed computing engine — it takes a big data problem, slices it into small chunks, and processes all those chunks in parallel across many machines simultaneously. When the work is done, it reassembles the results for you.

The thing that made Spark revolutionary is that it does this in memory (RAM) rather than on disk. Its predecessor, Hadoop MapReduce, wrote intermediate results to disk after every step. Spark holds everything in RAM across all steps — which is why it can be 10–100× faster for multi-step pipelines and iterative algorithms like machine learning.

It's also unified: one framework handles batch ETL, real-time streaming, SQL queries, machine learning, and graph analytics. You learn the API once and use it everywhere.

🍕 Analogy: You have a stadium order of 10,000 pizzas to make. One chef can't do it. Spark is the kitchen manager — it breaks the order into batches, hands each batch to a different chef, they all cook in parallel, and every pizza comes out at the same time. The bigger the kitchen (cluster), the faster you go.
Why not just use Python/pandas?

pandas is fantastic — for data that fits in your laptop's RAM (roughly <10 GB in practice). Once your dataset exceeds memory, pandas crashes, slows to a crawl, or forces you into ugly chunking workarounds.

Spark scales from one machine to thousands, and handles terabytes or petabytes without you changing a single line of logic. You write the transformation once; Spark figures out how to run it across the cluster.

2 Spark Architecture
The cluster model — how your code becomes distributed work
🏗 Understanding the cluster before you write a single line

A Spark application has three physical components: a Driver (your program's main process), a Cluster Manager (resource allocator), and Executors (the workers that actually run your code). The Driver never touches data directly — it coordinates. The Executors do all the heavy lifting on their local partitions of data.

This separation is deliberate. It lets Spark scale horizontally — you need more compute? Add more Executor machines. The Driver code doesn't change at all. And if an Executor fails mid-job, Spark automatically reruns its tasks on a healthy one, making it fault-tolerant by design.

👤 Driver

Your application's brain. It runs your main() function, converts your code into a DAG of tasks, and coordinates everything. It talks to the Cluster Manager to request resources, and talks to Executors to dispatch work and collect results. If the Driver dies, the job dies.

⚙ Cluster Manager

The resource broker. It decides how much CPU and RAM each application gets. Spark supports YARN (Hadoop clusters), Kubernetes (modern cloud-native), Mesos, and Standalone mode. The cluster manager is completely independent of Spark — you can swap it without touching your code.

🖥 Executor

A JVM process on a worker node that runs tasks and stores cached data. There are many executors per application — often dozens or hundreds. If an Executor dies mid-job, Spark automatically reruns its tasks on a healthy Executor. This fault tolerance is built in.

⚡ How a job actually executes — Job → Stage → Task

When you call an action (like count() or write()), Spark triggers a Job. The Driver looks at all the transformations you've chained together and builds a DAG — a directed acyclic graph of every computation step and how they depend on each other.

The DAG gets cut into Stages at every point where data needs to shuffle between machines (for example, a groupBy requires all rows with the same key to be on the same machine). Within a stage, every task is independent and can run in parallel.

A Task is the smallest unit — it processes exactly one Partition of data. The number of parallel tasks at any moment equals the number of partitions being processed. Partitions are typically 128–256 MB each.

💡 The Catalyst Optimizer is the secret weapon — it automatically rewrites your query plan into the most efficient physical execution, pushing filters down, reordering steps, and picking the best join strategy.
🔀 Shuffle — the most expensive operation in Spark

A shuffle happens when Spark needs to redistribute data across all executors — moving rows from their current machine to wherever they need to be for the next operation.

Shuffles involve network I/O, disk writes, and serialization. They are the main reason Spark jobs are slow. Minimising shuffles is the number one performance optimisation skill.

Operations that cause a shuffle:

  • groupBy() / groupByKey()
  • join() between two large DataFrames
  • repartition(n)
  • distinct() / dropDuplicates()
  • sortBy() / orderBy()
✓ No shuffle: map(), filter(), select(), withColumn(), coalesce() — these work independently on each partition.
3 RDD, DataFrame & Dataset
Three abstractions — choose the right one for the job
📦 Spark's three data abstractions — and why they exist

Spark has evolved over the years and introduced three different ways to represent distributed data. Each is built on top of the previous one and adds more structure. Beginners often get confused about which to use — the answer is almost always DataFrame, but knowing why helps you use it correctly.

RDD (Resilient Distributed Dataset) is the original low-level API — raw distributed collections you manipulate with functions. DataFrame adds a schema (named columns with types) so Spark can understand and optimise your data. Dataset adds compile-time type safety on top of DataFrame — but only in Scala and Java, not Python.

Legacy / Low-Level
🧱

RDD

Resilient Distributed Dataset. An immutable, partitioned collection of JVM objects distributed across the cluster. You write explicit map(), filter(), reduce() lambda functions to transform it.

Spark has no idea what's inside your RDD — it just runs your functions. That means no automatic optimisation from Catalyst. It also means poor performance for Python (data serialises between Python and JVM constantly).

🔧Like writing assembly code — total control, but you handle every detail yourself with no compiler help.

Use when: unstructured data, custom serialisation, legacy code

rdd = sc.textFile("hdfs://data.txt") words = rdd.flatMap(lambda line: line.split(" ")) counts = words.map(lambda w: (w, 1)) \ .reduceByKey(lambda a, b: a + b) counts.collect()
✨ Default Choice — Use This
📊

DataFrame

A distributed table with named, typed columns. Built on top of RDD but with a schema — Spark knows the structure, so the Catalyst Optimizer can rewrite and improve your query automatically.

It works identically whether you use the DataFrame API or write SQL — both go through the same optimizer and produce the same execution plan. DataFrames are also very efficient in Python because operations run in the JVM.

🗄Like writing SQL — you describe what you want, and Spark figures out the most efficient how.

Use for: 90%+ of all Spark work

df = spark.read.parquet("s3://bucket/sales/") df.filter("amount > 0") \ .groupBy("region") \ .agg(F.sum("amount").alias("revenue")) \ .show()
Scala / Java Only
🔒

Dataset

A typed DataFrame. You get all the Catalyst optimisation of DataFrames plus compile-time type safety. Instead of generic Row objects, you work with your own case classes (Scala) or POJOs (Java).

Python has no Dataset API — PySpark's DataFrames are effectively untyped. In Scala, Datasets are invaluable when you want the compiler to catch mistakes before your job runs in production.

🏗Like a typed ORM — the performance of SQL with the safety guarantees of a compiled, statically-typed language.

Use when: Scala/Java, need compile-time safety

// Scala only case class Sale(region: String, amount: Double) val ds: Dataset[Sale] = df.as[Sale] ds.filter(_.amount > 0) .groupByKey(_.region) .mapValues(_.amount) .reduceGroups(_ + _) .show()

4 Lazy Evaluation
Spark's most important concept — nothing runs until you say so
⏳ Why Spark is "lazy" — and why that's a superpower

When you write df.filter("age > 25").groupBy("city").agg(...), absolutely nothing happens. Spark just records your instructions as a logical plan. It's like writing a recipe — you're not cooking yet, just describing the steps.

Only when you call an action (like count(), show(), or write()) does Spark actually execute. And crucially, before executing, it hands the entire plan to the Catalyst Optimizer, which can see all the steps at once and figure out the smartest possible execution — combining steps, reordering them, pushing filters down to read less data from disk.

🗺 Analogy: You're planning a road trip. Writing down all your stops first (lazy) lets you calculate the most efficient route before you start driving. Taking each turn as you think of it (eager) means terrible detours and backtracking. Spark plans the whole route before driving a single mile.
⏳ Transformations — lazy, return a new DataFrame

These build up the execution plan without running anything. You can chain as many as you want — Spark accumulates them all into a DAG.

⚠ Never use collect() on large data. It pulls every row back to the Driver's memory. For a 500 GB dataset, that means 500 GB flows to one machine — instant OOM crash. Use write() to save output to storage instead.
What Catalyst actually does with your plan

Predicate Pushdown: If you filter after a join, Catalyst moves the filter before the join so less data is joined in the first place. When reading Parquet files, it pushes the filter all the way down to the file reader — entire row groups are skipped without being read into memory.

Projection Pruning: If you only use 3 of 50 columns, Catalyst ensures only those 3 columns are read from Parquet. In columnar formats, this is a huge speedup — you read a fraction of the data.

Join Reordering: Catalyst may reorder join operations to minimise intermediate dataset sizes. A smaller table joined first reduces how much data the subsequent join has to process.

Constant Folding: Expressions like 1 + 2 or lit(true) && col("x") > 0 are evaluated at compile time, not at runtime for every row.

💡 Use df.explain(True) to see the full query plan — Logical, Analysed, Optimised, and Physical. This is the most powerful debugging tool you have for performance issues.
5 SparkSession — Your Entry Point
Every Spark program starts here
🚪 The one object that gives you access to everything

SparkSession is the unified entry point to Spark. In old Spark (<2.0) you had separate objects — SparkContext for RDDs, SQLContext for DataFrames, HiveContext for Hive — and you had to manage all of them. SparkSession wraps them all into one clean object.

You get one SparkSession per application (though you can have multiple named ones). It gives you access to reading data, running SQL, accessing the Spark configuration, and creating DataFrames programmatically.

Creating a SparkSession
from pyspark.sql import SparkSession # Builder pattern — chain config calls spark = SparkSession.builder \ .appName("MySparkApp") \ # shows in Spark UI .master("local[*]") \ # local: use all CPU cores .config("spark.executor.memory", "4g") \ .config("spark.sql.adaptive.enabled", "true") \ .config("spark.sql.shuffle.partitions", "200") \ .getOrCreate() # reuse existing session if one exists # Access the underlying SparkContext (for RDD work) sc = spark.sparkContext # Always shut down cleanly when done spark.stop()
💡 .master("local[*]") runs Spark locally using all available CPU cores — perfect for development and testing. In production, you omit this and let the cluster manager set it, or pass --master as a spark-submit argument.

6 Reading & Writing Data
Getting data in and out of Spark — formats, options, and best practices
📂 Format matters enormously — choose wisely

The file format you choose affects performance more than almost any code-level optimisation. The key rule: use Parquet (or Delta Lake) for any serious production work. Here's why: Parquet is a columnar format, meaning each column is stored together rather than each row. When you only need 3 of 50 columns, Spark reads only those 3 columns from disk — potentially a 94% reduction in I/O. Parquet also stores statistics per row group (min/max values per column), enabling Spark to skip entire chunks of files that can't possibly match your filter.

Use CSV or JSON only for raw ingestion at the edge of your pipeline. Convert to Parquet immediately and work with Parquet from then on.

📚 Row vs Columnar: CSV stores data like a book (all of row 1, then all of row 2...). Parquet stores it like an encyclopedia index (all "names" together, all "ages" together). If you want everyone's age, an encyclopedia index lets you jump straight to that section — no skimming every row.
Reading data — all common formats
# ── CSV ────────────────────────────────────── df = spark.read \ .option("header", "true") \ # first row is header .option("inferSchema", "true") \ # auto-detect column types .option("sep", ",") \ # default is comma .option("nullValue", "NA") \ # treat "NA" as null .csv("s3://bucket/data.csv") # ── Parquet (preferred) ────────────────────── df = spark.read.parquet("s3://bucket/data/") # ── JSON (one object per line) ─────────────── df = spark.read.json("path/to/events.json") # ── Delta Lake ─────────────────────────────── df = spark.read.format("delta").load("s3://bucket/delta/")
# ── JDBC (relational databases) ────────────── df = spark.read \ .format("jdbc") \ .option("url", "jdbc:postgresql://host:5432/mydb") \ .option("dbtable", "orders") \ .option("user", "myuser") \ .option("password", "secret") \ .option("numPartitions", "8") \ # parallel reads .option("partitionColumn", "id") \ .option("lowerBound", "1") \ .option("upperBound", "1000000") \ .load() # ── With explicit schema (avoid inferSchema in prod) ── from pyspark.sql.types import StructType, StructField, StringType, DoubleType, TimestampType schema = StructType([ StructField("order_id", StringType(), nullable=False), StructField("amount", DoubleType(), nullable=True), StructField("created_at",TimestampType(), nullable=True), ]) df = spark.read.schema(schema).csv("s3://bucket/orders.csv")
⚡ Avoid inferSchema in production. Spark has to scan the entire dataset to figure out types — that's a full extra read. Always define your schema explicitly. It's also safer — a malformed file won't silently change your column types.
Writing data — modes, partitioning, file count
# ── Basic writes ───────────────────────────── df.write.mode("overwrite").parquet("s3://out/") df.write.mode("append").parquet("s3://out/") df.write.mode("ignore").parquet("s3://out/") # skip if exists df.write.mode("error").parquet("s3://out/") # default: fail if exists # ── Partition on disk (creates subdirectories) ─ # → out/date=2024-01-01/country=US/part-0000.parquet df.write \ .mode("overwrite") \ .partitionBy("date", "country") \ .parquet("s3://out/") # Readers can then filter: date=2024-01-01 → only that dir scanned
# ── Control number of output files ─────────── # coalesce: reduce partitions WITHOUT a shuffle (cheap) df.coalesce(1).write.parquet("out/") # one big file # repartition: redistribute WITH a shuffle (expensive) df.repartition(20).write.parquet("out/") # ── Write to Hive table ────────────────────── df.write.mode("overwrite").saveAsTable("mydb.orders") # ── JDBC write ─────────────────────────────── df.write \ .format("jdbc") \ .option("url", "jdbc:postgresql://host/db") \ .option("dbtable", "public.results") \ .option("user", "myuser") \ .option("password", "secret") \ .mode("append") \ .save()
💡 Partition your output for query performance. If downstream jobs always filter by date, partitioning by date means they only scan one folder instead of scanning everything. Partition by columns you filter on most — but avoid partitioning on high-cardinality columns like user_id, which would create millions of tiny folders.
7 DataFrame Transformations
The core operations for selecting, filtering, and shaping data
🔧 How to think about transformations

Every transformation takes a DataFrame and returns a new DataFrame — the original is never mutated. This immutability is by design: it makes Spark's fault recovery easy (replay from source) and enables safe parallel execution across many machines.

The key mental model: transformations are like SQL clauses. select() is your SELECT list, filter() is WHERE, groupBy()+agg() is GROUP BY, join() is JOIN, orderBy() is ORDER BY. If you can write it in SQL, you can write it as a transformation.

Selecting & reshaping columns
import pyspark.sql.functions as F # Select specific columns df.select("name", "age", "city") # Select with expressions df.select( F.col("name").alias("full_name"), (F.col("age") * 12).alias("age_months"), F.upper(F.col("city")).alias("city_upper"), F.when(F.col("age") >= 18, "adult") .otherwise("minor").alias("age_group"), ) # Add a new column (keep all existing columns) df.withColumn("tax", F.col("amount") * 0.2) df.withColumn("date", F.to_date(F.col("ts"))) df.withColumn("month", F.month(F.col("date"))) df.withColumn("parts", F.split(F.col("tags"), ",")) # Rename, drop columns df.withColumnRenamed("old_name", "new_name") df.drop("useless_col", "another_col")
F.when().otherwise() — the SQL CASE WHEN of Spark. Chain multiple .when() calls before .otherwise():
df.withColumn("tier", F.when(F.col("spend") > 10000, "platinum") .when(F.col("spend") > 5000, "gold") .when(F.col("spend") > 1000, "silver") .otherwise("standard") )
Handling nulls: Nulls are infectious in Spark — any operation on a null returns null. Use these to manage them:
# Replace nulls df.fillna("Unknown") # all string cols df.fillna({"city": "N/A", "age": 0}) # First non-null from multiple columns df.withColumn("phone", F.coalesce( F.col("mobile"), F.col("work"), F.lit("none") )) # Remove rows with any null in key cols df.dropna(subset=["user_id", "email"])
Filtering rows
# SQL string filter (convenient) df.filter("age > 25 AND city = 'London'") df.where("status IN ('active', 'pending')") # Column expression filter (safer — no SQL injection) df.filter(F.col("age") > 25) df.filter((F.col("age") > 25) & (F.col("city") == "London")) df.filter(F.col("city").isin("London", "Paris", "NYC")) df.filter(~F.col("city").isin("London", "Paris")) # NOT IN # Null checks df.filter(F.col("email").isNotNull()) df.filter(F.col("email").isNull()) # String matching df.filter(F.col("name").startswith("Al")) df.filter(F.col("email").endswith("@gmail.com")) df.filter(F.col("desc").contains("spark"))
# Regex filter df.filter(F.col("code").rlike("^[A-Z]{2}[0-9]{4}$")) # Date range filter df.filter(F.col("date") >= F.lit("2024-01-01")) df.filter(F.col("date").between("2024-01-01", "2024-12-31")) # Array / struct column filters df.filter(F.array_contains(F.col("tags"), "sale")) df.filter(F.size(F.col("items")) > 0) # Deduplication df.distinct() # exact row duplicates df.dropDuplicates(["user_id", "date"]) # key-based dedup # Sort / limit df.orderBy(F.col("amount").desc()) df.orderBy("date", F.col("amount").desc()) df.limit(100)
💡 Filter as early as possible. Even though Catalyst often pushes filters down automatically, explicitly filtering early in your code keeps your plan readable and ensures that in cases where Catalyst can't push the filter (e.g., through some UDFs), you've already reduced the data.

8 Aggregations
Collapsing rows into summaries — the groupBy + agg pattern
📊 How aggregations work under the hood

An aggregation takes many rows and produces fewer rows — for example, turning millions of individual transactions into one revenue total per region. In Spark, aggregations always involve a shuffle: rows with the same key must land on the same executor before they can be combined.

This happens in two phases. First, each executor does a partial aggregation on its local data (pre-aggregate before the shuffle to reduce data volume). Then the partially aggregated results are shuffled by key, and each executor finishes the aggregation on its slice. This two-phase approach is much more efficient than shipping all raw rows across the network.

groupBy + agg — the core pattern
# Basic aggregation df.groupBy("region").agg( F.count("*").alias("order_count"), F.sum("amount").alias("revenue"), F.avg("amount").alias("avg_order"), F.min("amount").alias("min_order"), F.max("amount").alias("max_order"), F.stddev("amount").alias("stddev"), F.countDistinct("user_id").alias("unique_users"), F.approx_count_distinct("user_id").alias("approx_uniq"), ) # Group by multiple columns df.groupBy("region", "category", "year").agg( F.sum("amount").alias("revenue") ) # HAVING equivalent — filter after aggregation df.groupBy("region").agg( F.sum("amount").alias("revenue") ).filter(F.col("revenue") > 100000)
# Collecting values into a list df.groupBy("user_id").agg( F.collect_list("item").alias("all_items"), # preserves dups F.collect_set("category").alias("categories"), # unique only F.first("name").alias("first_seen_name"), F.last("address").alias("last_address"), ) # Aggregate the whole DataFrame (no groupBy) df.agg( F.count("*").alias("total_rows"), F.sum("amount").alias("grand_total"), )
approx_count_distinct uses HyperLogLog internally — much faster than exact countDistinct for very large datasets with ~2% error. Use it when you need "roughly how many unique users" rather than an exact count.
Pivot tables & rollup
# ── Pivot: one column value becomes a column ── # Turn category values (Electronics, Clothing...) into columns df.groupBy("region") \ .pivot("category") \ # each category → its own column .sum("amount") # Explicit pivot values (faster — avoids extra scan) df.groupBy("region") \ .pivot("category", ["Electronics", "Clothing", "Food"]) \ .agg(F.sum("amount"))
# ── Rollup: hierarchical subtotals ─────────── # Like GROUP BY ROLLUP in SQL df.rollup("country", "region", "city") \ .agg(F.sum("revenue").alias("rev")) # → rows for: (country,region,city), (country,region), # (country), (null,null,null) = grand total # ── Cube: all combinations ─────────────────── df.cube("country", "region") \ .agg(F.sum("revenue")) # → every possible subset combination
9 Joins
Combining DataFrames — the most common source of performance problems
🔗 What actually happens when Spark joins two tables

A join combines rows from two DataFrames based on a matching condition. Sounds simple — but in a distributed system, the two DataFrames might have their matching rows on completely different machines. Spark must physically move data across the network to bring matching rows together. This is why joins are expensive, and why understanding how Spark handles them is crucial to writing fast code.

Spark chooses between different join strategies based on the size of the data. The most important thing you can do as a developer is tell Spark when one side is small enough to fit in memory — enabling a broadcast join that avoids the shuffle entirely.

Join code & patterns
# Basic join (same key name in both DataFrames) df_orders.join(df_customers, "user_id") # inner df_orders.join(df_customers, "user_id", "left") # left df_orders.join(df_customers, "user_id", "left_anti") # Different column names in each DataFrame df_orders.join( df_customers, df_orders.customer_id == df_customers.id, "left" ) # Join on multiple columns df1.join(df2, ["region", "date"], "inner") # Complex join condition df1.join(df2, (df1.user_id == df2.id) & (df1.date == df2.date) & (df1.amount > df2.threshold), "inner" )
# ★ BROADCAST JOIN — single most important optimisation ★ # If one DataFrame is small (<10MB), broadcast it. # Spark copies the small table to every executor's memory. # No shuffle needed — each executor joins locally. from pyspark.sql.functions import broadcast large_orders.join( broadcast(small_lookup_table), "country_code" ) # Auto-broadcast threshold (default 10MB) spark.conf.set( "spark.sql.autoBroadcastJoinThreshold", "50MB" )
✓ Broadcast join = no shuffle. For a table with 200 GB of orders joining to a 5 MB country code lookup, broadcast eliminates the shuffle entirely. This single change can take a join from 30 minutes to 2 minutes. Always use it for dimension tables, lookup tables, and configuration data.
Join strategies — what Spark does under the hood
Broadcast Hash Join

Small table is broadcast to every executor. Each executor builds a hash map from it and probes against its local partition of the large table. Zero shuffle. Fastest possible join. Triggered automatically for tables under autoBroadcastJoinThreshold, or explicitly with broadcast().

Sort-Merge Join

Default for two large tables. Both sides are shuffled by the join key so matching keys land on the same executor. Then both are sorted and merged like a zipper. Involves a full network shuffle — expensive but works for any size. Use when neither side fits in memory.

Cartesian / Nested Loop Join

Cross joins and some complex conditions fall back to nested loop — every row from left checked against every row from right. O(n × m) complexity. Even 1M × 1M rows = 1 trillion pairs. Avoid unless you genuinely need all combinations.

⚡ Skewed joins: If one join key value has far more rows than others (e.g., 90% of orders belong to one giant retailer), a single executor gets overwhelmed. Fix: enable spark.sql.adaptive.skewJoin.enabled=true (AQE handles it automatically in Spark 3+), or add a "salt" to the key to spread the load.
10 Window Functions
Aggregates that keep every row — ranking, running totals, lag/lead
🪟 The difference between groupBy and window functions

A groupBy aggregation collapses many rows into fewer rows. You lose the individual row detail. Sometimes you need the aggregated value alongside the original row — for example, "what is each employee's salary as a percentage of their department's total?" You need both the salary (individual) and the total (aggregate) on the same row.

That's exactly what window functions do. They compute an aggregate over a "window" of rows (defined by a partition and sort order) and attach the result back to each individual row without collapsing anything. Every input row produces exactly one output row.

🪟 Analogy: Imagine standing at each house on a street and asking "what's the average house price in my neighborhood?" You stay at your house — you don't get merged into a neighborhood summary. You just get a new column with the neighborhood average attached to your own row.
Defining a window & applying functions
from pyspark.sql.window import Window import pyspark.sql.functions as F # ── Define the window ───────────────────────── # partitionBy: like GROUP BY — defines which rows are "neighbors" # orderBy: the sort order within the partition # Rank within department, ordered by salary descending dept_window = Window.partitionBy("dept").orderBy(F.col("salary").desc()) # No partition — entire DataFrame is one window global_window = Window.orderBy("date") # ── Ranking functions ───────────────────────── df.withColumn("rank", F.rank().over(dept_window)) df.withColumn("dense_rank", F.dense_rank().over(dept_window)) df.withColumn("row_number", F.row_number().over(dept_window)) df.withColumn("percent_rank", F.percent_rank().over(dept_window)) df.withColumn("ntile", F.ntile(4).over(dept_window)) # quartiles
# ── Lag / Lead — look at neighboring rows ───── df.withColumn("prev_salary", F.lag("salary", 1).over(dept_window)) df.withColumn("next_salary", F.lead("salary", 1).over(dept_window)) df.withColumn("prev_default", F.lag("salary", 1, 0).over(dept_window)) # ── Aggregates over the window ──────────────── by_dept = Window.partitionBy("dept") # no orderBy = entire partition df.withColumn("dept_total", F.sum("salary").over(by_dept)) df.withColumn("dept_avg", F.avg("salary").over(by_dept)) df.withColumn("dept_max", F.max("salary").over(by_dept)) df.withColumn("pct_of_dept", F.col("salary") / F.sum("salary").over(by_dept)) # ── Running totals (cumulative sum) ─────────── cumulative = Window.partitionBy("dept").orderBy("date") \ .rowsBetween(Window.unboundedPreceding, Window.currentRow) df.withColumn("running_total", F.sum("amount").over(cumulative))

Classic use case: Top-N rows per group
# Get the top 3 highest-paid employees per department # Step 1: assign row_number within each partition w = Window.partitionBy("dept").orderBy(F.col("salary").desc()) ranked = df.withColumn("rn", F.row_number().over(w)) # Step 2: filter to keep only top 3 per partition ranked.filter(F.col("rn") <= 3).drop("rn") # This pattern replaces complex self-joins or subqueries. # Use row_number (not rank) if you want exactly N rows — rank may give more if there are ties.
Frame specification — controlling which rows are "in the window"

When you include orderBy() in a window, Spark adds a default frame of unboundedPreceding to currentRow (all rows from the start to now). This makes sum() compute a cumulative sum. You can override this with rowsBetween() or rangeBetween().

rowsBetween: physical row offsets from current row.
rangeBetween: logical value range based on the ORDER BY column.

# Moving average: current row ± 2 rows (5-row window) rolling = Window.partitionBy("region").orderBy("date") \ .rowsBetween(-2, 2) df.withColumn("moving_avg", F.avg("revenue").over(rolling)) # All previous rows up to current (cumulative) cum = Window.orderBy("date") \ .rowsBetween(Window.unboundedPreceding, Window.currentRow) df.withColumn("cumulative_sum", F.sum("amount").over(cum)) # All rows in the partition (no row boundaries) full = Window.partitionBy("dept") df.withColumn("dept_total", F.sum("salary").over(full))
11 Structured Streaming
Real-time data processing with the same DataFrame API you already know
🌊 The mental model: an infinite table

The key insight behind Structured Streaming is beautifully simple: treat your live stream of events as an infinite table. New events are just new rows being appended to that table continuously. Your query — which looks identical to a batch DataFrame query — runs continuously against this growing table.

Under the hood, Spark executes your query as a series of small micro-batches — triggered every few seconds (or continuously, in the newer continuous mode). Each micro-batch processes only the new data that arrived since the last one ran. State for aggregations is maintained across micro-batches automatically.

This design means you write one set of data transformation logic and run it in both batch and streaming contexts — a huge operational win.

📋 Analogy: Imagine a spreadsheet that keeps having new rows added at the bottom automatically. Your SUM formula at the top updates every few seconds to include the latest rows. You didn't have to write any "streaming" logic — the formula is the same, the spreadsheet just keeps growing.
Reading a stream & writing output
from pyspark.sql import SparkSession import pyspark.sql.functions as F spark = SparkSession.builder.appName("Streaming").getOrCreate() # ── Read stream from Kafka ───────────────────── raw = spark.readStream \ .format("kafka") \ .option("kafka.bootstrap.servers", "broker:9092") \ .option("subscribe", "events") \ .option("startingOffsets", "latest") \ .load() # ── Read stream from a directory (new files drop in) ── file_stream = spark.readStream \ .schema(schema) \ # schema required for file streams .option("maxFilesPerTrigger", 1) \ .parquet("s3://bucket/landing/") # ── Apply transformations (identical to batch!) ── events = raw.select( F.from_json(F.col("value").cast("string"), schema).alias("d") ).select("d.*") result = events \ .filter(F.col("amount") > 0) \ .groupBy("region") \ .agg(F.sum("amount").alias("revenue"))
# ── Write stream to sink ────────────────────── query = result.writeStream \ .outputMode("update") \ # see output modes below .format("console") \ # dev only .trigger(processingTime="30 seconds") \ .option("checkpointLocation", "/tmp/checkpoint/my-job") \ .start() query.awaitTermination() # block until manually stopped # ── Write to Kafka sink ─────────────────────── result.select( F.col("region").alias("key"), F.to_json(F.struct("region", "revenue")).alias("value") ).writeStream \ .format("kafka") \ .option("kafka.bootstrap.servers", "broker:9092") \ .option("topic", "output-topic") \ .option("checkpointLocation", "/tmp/cp/kafka-out") \ .outputMode("update") \ .start() # ── Write to Delta Lake sink ────────────────── result.writeStream \ .format("delta") \ .option("checkpointLocation", "/tmp/cp/delta-out") \ .outputMode("append") \ .start("s3://bucket/output-delta/")
Output modes explained
append (default) — Only newly completed rows are written. No row already written is ever updated. Required for queries without aggregations, or aggregations with watermarks. Best for: event logs, audit trails, write-once sinks.
update — Only rows that changed since the last trigger are written. Aggregated values get updated each micro-batch. Best for: dashboards, real-time counters, key-value stores.
complete — The entire result table is rewritten every trigger. Only works with aggregations. Expensive for large outputs. Best for: small result sets like "top 10" leaderboards that must always show the full current state.
⚠ Watermarks — handling late data

In real-world streaming, events arrive out of order. A purchase at 3:00 PM might reach your pipeline at 3:15 PM due to network delays or mobile app batching. Without watermarks, Spark would have to hold state for all past time windows forever — not feasible.

A watermark tells Spark: "wait up to N minutes for late data, then close the window." Any events arriving more than N minutes late are dropped.

# Allow events up to 10 minutes late events \ .withWatermark("event_time", "10 minutes") \ .groupBy( F.window("event_time", "5 minutes"), # 5-min tumbling window "region" ) \ .agg(F.count("*").alias("event_count")) # Sliding window: "5 minutes every 1 minute" F.window("event_time", "5 minutes", "1 minute")
⚠ Always set checkpointLocation. This is how Spark tracks exactly which data has been processed. Without it, restarting your streaming job will either reprocess everything from the beginning or lose data. It must point to reliable, durable storage (HDFS or S3).
12 Optimization & Tuning
Where jobs go slow and exactly how to fix them
⚡ The 80/20 of Spark performance

Most Spark performance problems come from a small set of root causes: too many shuffles, data skew (one partition is 100× larger than others), too many tiny partitions or too few large ones, not caching reused DataFrames, and failing to broadcast small lookup tables. Learn to spot these in the Spark UI and you can fix 80% of slow jobs.

The Spark UI (port 4040 during a job, History Server afterwards) is your primary debugging tool. The Jobs tab shows which stages are slow. The Stages tab shows task distribution — if one task takes 10× longer than the rest, you have data skew. The SQL tab shows the query plan visually.

💾 Caching — avoid recomputing the same data

By default, Spark recomputes a DataFrame from scratch every time it's used. If you use the same intermediate result in multiple places — for example, a cleaned and joined dataset that feeds both an aggregation and a write — Spark reads and processes the source data twice.

Calling cache() materialises the DataFrame into executor memory after its first computation. Every subsequent use reads from memory — instant. Only cache what you genuinely reuse; caching consumes memory and may evict other partitions.

from pyspark import StorageLevel # Default: in-memory only, JVM objects (fast, uses most RAM) df.cache() # shortcut for MEMORY_ONLY # Spill to disk if memory is full (safer for large DFs) df.persist(StorageLevel.MEMORY_AND_DISK) # Serialised (smaller memory footprint, slightly slower access) df.persist(StorageLevel.MEMORY_AND_DISK_SER) # Disk only (slowest, but doesn't use memory at all) df.persist(StorageLevel.DISK_ONLY) # ⚠ Cache is LAZY — it only materialises when an action runs # To force materialisation immediately: df.cache() df.count() # this action triggers the actual caching # Always unpersist when you're done df.unpersist() # When to cache — the rule of thumb: # Cache if a DataFrame is used 2+ times AND # it takes more than a few seconds to compute
🔀 Partitioning — the unit of parallelism

Each partition is processed by exactly one task on one executor thread. The number of partitions directly determines how many tasks run in parallel. Get this wrong and you leave CPU cores idle or overwhelmed.

Too few partitions: most executors sit idle while a few crunch large chunks. Too many partitions: scheduling overhead, small tasks finish faster than they can be launched.

Target: 128–256 MB per partition, and 2–4× as many partitions as the total number of CPU cores in your cluster.

# Check current partition count df.rdd.getNumPartitions() # repartition(): full shuffle, creates even distribution # Use when: too few partitions, or very skewed df = df.repartition(200) df = df.repartition(200, "user_id") # hash by column # coalesce(): reduce partitions WITHOUT a shuffle # Much cheaper than repartition — use after filter cuts data df = df.coalesce(50) # The default shuffle partition count (after joins, groupBy) # Default is 200 — wrong for almost every cluster spark.conf.set("spark.sql.shuffle.partitions", "400") # Or let AQE figure it out automatically (see below) spark.conf.set("spark.sql.adaptive.enabled", "true")
🤖 Adaptive Query Execution (AQE) — enable this immediately

AQE (available in Spark 3.0+) is a runtime optimizer that replans your query while it's executing, using actual statistics collected mid-job. Before AQE, Spark had to make all decisions upfront based on estimates that were often wildly wrong.

Three things AQE does automatically:

  • Coalesce small shuffle partitions: If your shuffle produces 200 tiny partitions, AQE merges them into fewer larger ones — reducing task scheduling overhead.
  • Convert sort-merge joins to broadcast joins: If one side turns out much smaller than estimated at runtime, AQE upgrades the join strategy on the fly.
  • Handle skewed joins: If one partition is much larger than others (data skew), AQE automatically splits it and replicates the other side to match.
# ── Enable AQE (should be default in Spark 3.2+) ───── spark.conf.set("spark.sql.adaptive.enabled", "true") # Auto-coalesce shuffle partitions spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true") spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "128MB") # Auto-broadcast at runtime (overrides static threshold) spark.conf.set("spark.sql.adaptive.autoBroadcastJoinThreshold", "30MB") # Handle skewed joins automatically spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true") spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB") # What AQE considers "skewed" # A partition is skewed if it's > N * median partition size spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")
⚠ Common pitfalls & how to fix them
  • collect() on large data: Pulls all data to driver → OOM crash. Fix: use write() to storage instead. Only use collect() on small aggregated results.
  • Data skew: One partition has 100× more rows than others → one task takes forever while all other executors sit idle. Fix: enable AQE skew join handling, or manually salt the key.
  • Python UDFs: Each row serialises from JVM to Python process and back → 10–100× slower than built-in functions. Fix: replace with built-in F.* functions. If custom logic is necessary, use Pandas UDFs (vectorised) instead.
  • Too many small files on write: Hundreds of 1KB Parquet files → slow reads later. Fix: coalesce before writing, or use Delta Lake's OPTIMIZE.
  • shuffle.partitions=200 always: For a 1 TB job, 200 partitions means 5 GB each — executors OOM. For a 100 MB job, 200 partitions means 0.5 MB tasks — pure overhead. Fix: tune to cluster size, or enable AQE.
  • Not caching reused DataFrames: Every branch recomputes from scratch. Fix: cache() before the fork point.

13 MLlib — Distributed Machine Learning
Training models on data too large for a single machine
🧠 When to use Spark for ML (and when not to)

MLlib is Spark's built-in machine learning library. It runs ML algorithms in a distributed fashion across your cluster — the training data is split across executors, partial computations are done locally, and results are combined. This enables training on datasets that don't fit on one machine.

When Spark ML makes sense: your training data is large (tens of GB to TB), your feature engineering is already happening in Spark, or you need to run inference on very large datasets at scale.

When it doesn't: if your data fits in RAM on one machine, scikit-learn or XGBoost will be simpler, faster to iterate on, and have more algorithm options. Don't add Spark complexity unless you genuinely need the scale.

The Pipeline pattern — the right way to build ML in Spark

Spark ML is structured around Pipelines — ordered sequences of Transformers and Estimators. A Transformer applies a transformation (like scaling features). An Estimator fits on data and produces a Transformer (like training a model produces a fitted model that can transform new data).

You chain them into a Pipeline and call pipeline.fit(train_df). The pipeline fits all stages in order. The resulting PipelineModel applies all stages when you call transform() — on test data or production data. Feature engineering is baked into the model, so training and inference are always consistent.

  • Consistent preprocessing between train and inference
  • One fit() call runs the whole pipeline
  • Save/load the entire pipeline including fitted encoders
  • Hyperparameter tuning with CrossValidator works on the full pipeline
from pyspark.ml import Pipeline from pyspark.ml.feature import ( StringIndexer, OneHotEncoder, VectorAssembler, StandardScaler ) from pyspark.ml.classification import RandomForestClassifier from pyspark.ml.evaluation import BinaryClassificationEvaluator # 1. Encode categorical column to integer index indexer = StringIndexer(inputCol="category", outputCol="category_idx") # 2. One-hot encode the index encoder = OneHotEncoder(inputCol="category_idx", outputCol="category_vec") # 3. Combine all feature columns into one vector assembler = VectorAssembler( inputCols=["age", "income", "category_vec"], outputCol="features_raw" ) # 4. The model rf = RandomForestClassifier( featuresCol="features", labelCol="label", numTrees=100, maxDepth=8 )
# Assemble the pipeline pipeline = Pipeline(stages=[indexer, encoder, assembler, scaler, rf]) # Train/test split train_df, test_df = df.randomSplit([0.8, 0.2], seed=42) # Fit the pipeline on training data model = pipeline.fit(train_df) # Apply to test data (all pipeline stages run in order) predictions = model.transform(test_df) # predictions now has columns: features, rawPrediction, probability, prediction # Evaluate evaluator = BinaryClassificationEvaluator(labelCol="label") auc = evaluator.evaluate(predictions) print(f"AUC-ROC: {auc:.4f}") # Save the entire pipeline (fitted — includes encoders) model.save("s3://bucket/models/my-model-v1") # Load and use later for inference from pyspark.ml import PipelineModel loaded_model = PipelineModel.load("s3://bucket/models/my-model-v1") batch_predictions = loaded_model.transform(new_data_df)

Where Spark Fits in the Bigger Picture

Spark & Apache Kafka

Once you're comfortable with Spark, you'll quickly encounter Apache Kafka — the distributed event streaming platform that is Spark Streaming's most common data source in production.

Kafka acts as a durable, high-throughput message bus. Applications write events (clicks, purchases, sensor readings) to Kafka topics at millions per second. Spark Structured Streaming reads from those topics, processes the events in near-real-time, and writes enriched results to a data warehouse, database, or back to another Kafka topic.

The two are genuinely complementary: Kafka is excellent at ingesting and buffering streams reliably. Spark is excellent at complex computation on those streams — joins, aggregations, ML inference, enrichment with reference data. Together they form the backbone of most modern real-time data platforms.
