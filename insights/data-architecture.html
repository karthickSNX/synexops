Section 00
Storage Architecture — Lake · Warehouse · Lakehouse

Before you pick a tool, you need to understand where your data actually lives and the fundamental trade-offs of each storage paradigm. Data Lake, Data Warehouse, and Data Lakehouse are not interchangeable — they solve different problems. Getting this wrong means rebuilding your architecture in 18 months. This section also covers the Medallion Architecture (Bronze → Silver → Gold), the industry-standard pattern for organizing data quality layers inside a lake or lakehouse.

Data Lake
🏞️ Raw, Schema-on-Read

A data lake is a massive, cheap object store (S3, GCS, ADLS) where you dump raw data in any format — JSON, CSV, Parquet, images, logs — without defining a schema upfront. Schema is applied only when you query it. Incredibly flexible; potentially chaotic without discipline.

💡
Plain English A data lake is like a giant warehouse where you throw everything in boxes labeled only with a date. Cheap to store, but finding anything specific requires knowing where to look. Power comes from the breadth of data you've preserved — even data whose future use you didn't know at collection time.
Best for: raw ingestion, ML training data, event logs, unstructured data, low-cost long-term storage
Risks: "data swamp" without governance — no schema enforcement, no quality checks, no lineage
S3 / GCS / ADLSAny formatCheap storage
Data Warehouse
🏛️ Structured, Schema-on-Write

A data warehouse stores structured, modeled, analytics-ready data. Schema is strictly enforced at write time. Optimized for fast SQL queries with columnar storage and MPP (Massively Parallel Processing). The classic BI layer — fast, governed, expensive per TB.

💡
Plain English A data warehouse is like a well-organized library. Every book has a proper place, every shelf is labeled, and the catalog tells you exactly where to find anything. Queries are fast because everything is organized. But getting a new book on the shelf takes work — you have to catalog and organize it first.
Best for: BI dashboards, structured SQL analytics, governed reporting, financial data
Risks: expensive at scale, rigid schema makes iteration slow, poor support for unstructured data or ML
SnowflakeBigQueryRedshift
Data Lakehouse
🔮 Best of Both Worlds

A lakehouse combines the cheap open storage of a data lake with the ACID transactions, schema enforcement, and query performance of a warehouse — on the same data, using open formats. No data duplication. One platform for BI, ML, and streaming.

💡
Plain English The lakehouse is like building a modern library on top of that giant warehouse — you keep all the raw boxes, but now there's an index, shelving system, and librarians that enforce quality. Same data, accessible both as raw files (for ML) and as structured tables (for BI). This is where the industry is heading.
Best for: unified platform for BI + ML, reducing data duplication, modern cloud-native architectures
Trade-off: more complexity to operate than a pure warehouse; requires Delta Lake, Iceberg, or Hudi
DatabricksMicrosoft FabricDelta LakeIceberg

So how does a Lakehouse actually work under the hood?
Understanding the layer stack — before diving into the tools

A Data Lakehouse is not a single product you buy — it is an architectural pattern assembled from layers. The foundation is always cheap object storage (S3, GCS, ADLS). Sitting on top of that is a Table Format — Delta Lake, Apache Iceberg, or Apache Hudi. This table format is what transforms a folder of raw Parquet files into a real database-like table: it adds ACID guarantees, time travel, schema enforcement, and MERGE/UPSERT/DELETE. On top of the table format, a Query Engine (Spark, Trino, Flink) reads and writes the data. And at the top, your BI tools, ML pipelines, and dashboards consume it.

Delta Lake, Iceberg, and Hudi all occupy Layer 2. They all solve the same problem — making cheap object storage behave like a proper transactional database — but they make different trade-offs. The comparison cards below show exactly where they differ. The sections that follow in this guide (Spark, Kafka, dbt, Orchestration) operate at Layers 3 and above, building on whichever table format you choose here.

Delta Lake
⚡ Parquet + Transaction Log

Built by Databricks. Adds a _delta_log folder alongside your Parquet files that records every transaction as JSON — giving you ACID guarantees, time travel, and MERGE/UPSERT/DELETE on a plain object store. If you're on Databricks or an all-Spark stack, this is your default, full stop.

ACID transactions — safe concurrent reads and writes, no corrupt partial writes
Time travel — VERSION AS OF 5 or TIMESTAMP AS OF '2024-01-01'
Schema enforcement — rejects writes that don't match the table schema
OPTIMIZE + ZORDER — compact small files and co-locate related data for fast queries
Databricks-centric — best supported on Spark; other engines (Trino, Flink) have improving but lagging support
DatabricksSparkACIDTime Travel

Choose Delta when: you're on Databricks, or your entire stack is Spark-based and you don't need multi-engine portability.

Apache Iceberg
🔓 Open Standard, Multi-Engine

Designed from the ground up to be engine-agnostic. The same Iceberg table can be read and written by Spark, Flink, Trino, Athena, BigQuery, and Snowflake — no vendor lock-in. Its killer feature is partition evolution: change your partitioning strategy without rewriting any data.

True multi-engine — Spark, Trino, Flink, Athena, BigQuery, Snowflake all read natively
Partition evolution — change partition scheme without rewriting files. Delta's traditional partitioning requires rewrite; however Liquid Clustering (Delta 3.0+) achieves flexible layout evolution without full rewrites.
Hidden partitioning — partition on derived columns (year/month/day from timestamp) transparently
Row-level deletes — supports GDPR right-to-erasure without full partition rewrites
More complex to configure and tune than Delta if you're already on Databricks
Multi-engineOpen standardTrino / Athena

Choose Iceberg when: you use multiple query engines, want cloud portability, or are building a new architecture not tied to Databricks.

Apache Hudi
🔄 Upserts at Sub-Minute Latency

Hudi (Hadoop Upserts Deletes and Incrementals) was built at Uber for streaming upserts at very high frequency. Where Delta and Iceberg optimize for batch reads, Hudi optimizes for write-heavy workloads that need fresh data fast — think CDC pipelines landing millions of rows per minute.

Copy-on-Write (CoW) — rewrites entire file on update. Slow writes, very fast reads. Good for read-heavy BI.
Merge-on-Read (MoR) — logs deltas, compacts lazily. Fast writes, slightly slower reads. Good for streaming CDC.
Incremental pull — consumers can ask "give me only what changed since timestamp X" — very efficient for downstream pipelines
Smaller ecosystem — less tooling support than Delta or Iceberg; primarily Spark + Flink
CDC pipelinesHigh-frequency writesIncremental pull

Choose Hudi when: you have high-frequency CDC or streaming upserts and need sub-minute data freshness. Used at Uber, Robinhood, Amazon.

Quick Pick 🏢 On Databricks / all Spark → Delta Lake 🔓 Multi-engine / cloud portability → Apache Iceberg ⚡ High-frequency CDC / streaming upserts → Apache Hudi 📊 Pure analytics reads, no updates → Parquet (no table format needed)
Medallion Architecture
🥇 Bronze → Silver → Gold

The Medallion Architecture is the standard pattern for organizing data quality layers inside a lake or lakehouse. Three zones, each progressively more refined. Pioneered by Databricks, now universal. Your raw data is immutable (Bronze), your cleaned data is trusted (Silver), and your business-ready data is aggregated (Gold).

💡
Plain English Think of a coffee shop. Bronze = the raw coffee beans that just arrived (unprocessed, kept as-is). Silver = beans that are roasted, sorted, and quality-checked (clean, validated, deduplicated). Gold = the final espresso shot ready to serve (aggregated, joined, business-ready for dashboards or ML). You never throw away the beans — that's your audit trail.
🥉 Bronze
Raw landing zone. Exact copy of source data. Never modified. Append-only. JSON/CSV/Avro as-is. Think of it as your immutable audit log. Reprocess from here if anything goes wrong.
🥈 Silver
Cleaned & conformed. Nulls handled, types cast, deduped, validated. Joined where needed. Still row-level granularity — no aggregation yet. Source of truth for analysts.
🥇 Gold
Business-ready aggregates. Fact + dimension tables (Kimball), OBT for ML, domain-specific marts. What BI tools and ML models actually consume. Built on top of Silver with dbt or Spark.

s3://data-platform/ ├── bronze/ # raw, immutable, append-only │ ├── orders/2024/01/ # partitioned by ingest date │ └── users/2024/01/ ├── silver/ # cleaned, validated, conformed │ ├── orders/ # typed, deduped, nulls handled │ └── users/ └── gold/ # aggregated, business-ready ├── fct_orders/ # Kimball fact tables ├── dim_customers/ # dimension tables └── customer_ltv/ # ML features, aggregates
🗂️ Data Governance, Lineage & Catalog

Three interconnected disciplines that keep your data trustworthy and findable at scale. Often overlooked early, always regretted later.

Data Governance
The policies, standards, and accountability structures that ensure data is accurate, secure, compliant, and used appropriately. Who owns each dataset? What's the retention policy? Who can access PII? Governance is organizational as much as technical. Tools: Apache Atlas, Collibra, Alation, Purview.
Data Lineage
Tracks where every column came from and what it flows into — source system → Bronze → Silver → Gold → dashboard. When a number looks wrong, lineage tells you exactly which upstream transformation or source to investigate. Tools: DataHub (column-level lineage auto-detected from dbt), OpenLineage, Marquez.
Data Catalog
An inventory of all your data assets — tables, columns, owners, descriptions, PII tags, quality scores, and usage stats. Think of it as Google for your internal data. Without a catalog, analysts spend 40% of their time finding and understanding data before they can use it. Tools: DataHub, Atlan, Alation, dbt docs.
🌐 Distributed Systems & Scalability Concepts

Data engineering is distributed systems engineering. These concepts underpin every tool you use.

ConceptWhat it means
ScalabilitySystem handles growing workload by adding resources. Horizontal = more nodes. Vertical = bigger nodes.
ElasticityAutomatically scales up and down with demand. Cloud-native — Snowflake warehouses, Spark autoscaling, serverless functions.
Fault ToleranceSystem continues operating when parts fail. Kafka replicates partitions across brokers. Spark reruns failed tasks. HDFS stores 3 copies.
PartitioningSplitting data into segments for parallel processing and faster queries. Horizontal (row-based) or vertical (column-based).
ShardingDistributing data across multiple database servers to avoid a single bottleneck. Common in operational DBs (MongoDB, Cassandra).

Plain English — Scalability vs Elasticity vs Fault Tolerance A scalable highway can add more lanes to handle rush hour. An elastic highway magically gains lanes during rush hour and shrinks them at 2am. A fault tolerant highway reroutes traffic automatically when a lane closes — drivers don't even notice.

The Iceberg advantage that matters most: Iceberg tracks columns by a hidden integer ID, not by name. This means renaming a column in Iceberg is a pure metadata operation and never breaks readers. 🗜️ The Small File Problem — Deep Dive & Compaction Strategies

Small files are the silent killer of data lake performance. A Spark job that writes one file per partition per micro-batch will generate millions of files over weeks. Each file requires a separate S3 GET request, a Parquet footer read, and a metadata entry — at scale this destroys query performance and can OOM your Spark driver just opening the table.

Why It Happens
Streaming micro-batches — Spark Structured Streaming writes one file per partition per trigger interval. 5-minute batches × 200 partitions × 24h = 57,600 files/day.
High-cardinality partitioning — partitioning by user_id or UUID creates one directory per user. A table with 10M users = 10M partition directories.
INSERT-heavy workloads — each Spark job appending small batches accumulates files fast. Target: 100MB–1GB per Parquet file.
Detection
SQL — find small file partitions
-- Delta-specific: DESCRIBE DETAIL only works on Delta tables DESCRIBE DETAIL prod.orders.fct_orders; -- numFiles > 10000 and avgFileSize < 20MB → compaction needed -- Vanilla Parquet: use dbutils.fs.ls('/path/') or os.listdir() instead -- Spark: count files per partition from pyspark.sql import functions as F df = spark.read.format("delta").load(path) spark.sql(""" SELECT partition_date, COUNT(*) AS file_count, ROUND(SUM(size)/1e6, 1) AS total_mb, ROUND(AVG(size)/1e6, 2) AS avg_file_mb FROM (DESCRIBE DETAIL delta.`/path/to/table`) GROUP BY 1 ORDER BY file_count DESC """)
Compaction Strategies — Pick One Per Table Format
Delta Lake — OPTIMIZE
SQL / Python
-- Compact small files in a partition range OPTIMIZE prod.orders.fct_orders WHERE order_date BETWEEN '2024-01-01' AND '2024-01-31'; -- ZORDER co-locates related data for fast filters OPTIMIZE prod.orders.fct_orders ZORDER BY (customer_id, order_date); -- Auto-Optimize (Databricks only) ALTER TABLE fct_orders SET TBLPROPERTIES ( 'delta.autoOptimize.optimizeWrite' = 'true', 'delta.autoOptimize.autoCompact' = 'true' );

Raw Spark — coalesce / repartition
PySpark — as pipeline stage
# coalesce: reduces partitions, no shuffle # Use when current count is too high target_files = 20 # ~500MB each for 10GB table (df .coalesce(target_files) .write.mode("overwrite") .parquet(output_path)) # repartition: full shuffle, even distribution # Use when data is skewed across partitions (df .repartition(40, "order_date") .write.mode("overwrite") .parquet(output_path)) # Rule: coalesce is cheap, repartition is correct

Run compaction as a dedicated pipeline stage. Don't compact inline during your write — it slows your main pipeline. Instead, run an Airflow DAG that fires an OPTIMIZE/rewrite job daily at a low-traffic time, scoped to only the partitions written in the last 24h. This keeps your main pipeline fast and your files healthy.

⏱️ Processing Paradigms

The first decision in any pipeline: how often does it run, and how fresh must the results be? Each paradigm is a different trade-off between cost, complexity, and latency.

Batch Processing — processes large chunks of data on a schedule (hourly/daily). Simple, cost-efficient, proven at scale. 80% of pipelines are batch. Tools: Spark, dbt, Glue.
Micro-batch — runs every 30–60 seconds. Spark Structured Streaming default. Sweet spot between freshness and complexity — most "near real-time" needs are met here.
Real-Time Processing — event-by-event, sub-second latency. Apache Flink, Kafka Streams. Reach for this only when the business genuinely needs millisecond response — fraud detection, live leaderboards.
💡
Workflow Scheduler (Airflow, Dagster, Prefect) is what triggers your batch pipelines — it's the conductor. It monitors for failures, retries tasks, and sends alerts. Without a scheduler, your pipelines are just scripts someone runs manually.
ETL vs ELT
ETL
ETL (Extract, Transform, Load) — transform before loading. Legacy pattern — used when your warehouse was expensive and storage was scarce. Data was cleaned on the way in.
ELT
ELT (Extract, Load, Transform) — transform after loading. Modern default. Load raw data first, then transform with dbt inside the warehouse. Raw data is always preserved.
💡
Rule of thumb: if you're using a cloud DW (Snowflake, BigQuery, Redshift) — use ELT. The warehouse has the compute, use it.

🔁 Idempotency — The Golden Rule

A pipeline is idempotent when running it twice produces the same result as running it once. This is non-negotiable in production.

💡
Plain English Imagine a vending machine that charges you once, whether you press the button one time or ten. An idempotent pipeline works the same way — it doesn't matter if it runs once or five times, the result is always the same. Without this, a retry after a crash doubles your data.
⚠️
Never use plain INSERT without deduplication. Use MERGE, partition overwrite, or INSERT OVERWRITE so re-runs don't double your data.
🏛️ Lambda vs Kappa Architecture

Two competing schools of thought on how to build a streaming + batch system.

Lambda
Two paths: a slow batch layer for accuracy + a fast speed layer for low latency. They merge at serving. Ops-heavy. Hadoop-era design.
Kappa
Everything is a stream. Replay your Kafka log to reprocess history. One codebase, one system. Kafka + Flink or Spark Structured Streaming.
Most modern teams adopt Kappa or a simplified Lambda where the "batch layer" is just scheduled Spark jobs.

File Formats

Every data engineer eventually asks: "should I use Parquet or Avro for this?" The answer depends entirely on your use case. File format is not a trivial choice — it determines query speed, storage cost, schema evolution capability, and which tools can read your data. This section covers raw file formats: Parquet, Avro, ORC, JSON, CSV. For table formats (Delta Lake, Iceberg, Hudi) that sit on top of these, see Section 00.

Parquet

Columnar binary format — stores data column-by-column instead of row-by-row. This means if you query SELECT amount FROM orders, it only reads the amount column, skipping everything else. Massive I/O savings. Snappy/ZSTD compression built in. The industry default for analytics data.

💡
Plain English Think of a book index. Instead of reading every page to find mentions of "revenue," you jump straight to the revenue chapter. Parquet does this with data — query only the columns you need, skip the rest. A 1TB table with 100 columns? Read just the 2 you need, load ~20GB instead.
ColumnarCompressedAnalytics default

Pro tip: target 100MB–1GB per file. Millions of tiny Parquet files will destroy performance.

Avro

Row-based binary format. Schema is stored in the file header as JSON. The killer feature: schema evolution — you can add, rename, or remove fields without breaking existing consumers, as long as you follow compatibility rules. Kafka's native serialization format.

Streaming nativeSchema evolution

Use for: Kafka topics, CDC events, write-heavy workloads. Pair with Confluent Schema Registry.

Delta Lake / Iceberg / Hudi

These are table formats, not raw file formats — they sit on top of Parquet and add ACID transactions, time travel, and schema enforcement. They're how you build a Lakehouse on an object store. Join Strategies
🔀 How Spark Moves Data Across a Cluster

Every join in Spark causes one of three things to happen. Understanding which strategy is chosen — and why — is the difference between a 5-minute job and a 5-hour job. Spark picks automatically, but it picks wrong often enough that you need to know how to override it.

StrategyHow it worksNetwork costWhen Spark uses it
Broadcast Sends the entire small table to every executor. Each executor joins locally — zero shuffle. None One side < spark.sql.autoBroadcastJoinThreshold (default 10MB)
Shuffle Hash Both tables are hashed on the join key and shuffled so matching keys land on the same node, then hash-joined in memory. High One side large, other medium; AQE may select dynamically
Sort-Merge Both sides shuffled by key, sorted, then merged in a streaming pass. Handles any size — neither side needs to fit in memory. Very high Default for large-large joins. Most expensive, most reliable.
💡
Plain English Broadcast = photocopy the small table and hand it to every worker — no network traffic during the join. Shuffle Hash = sort everyone into rooms by their ID, then join inside each room — getting into rooms costs time. Sort-Merge = like Shuffle Hash but sort within each room first — slowest, but works even if rooms are enormous.
PySpark — forcing join strategies
from pyspark.sql.functions import broadcast # Force broadcast — fastest for small dims (<100MB) orders.join(broadcast(dim_status), "status_id") # Force sort-merge (large-large joins) orders.hint("merge").join(customers, "customer_id") # See what strategy Spark actually chose: df.explain("formatted") # Look for: BroadcastHashJoin / ShuffledHashJoin # / SortMergeJoin in the plan # Raise broadcast threshold (default 10MB) spark.conf.set( "spark.sql.autoBroadcastJoinThreshold", "104857600" )
Memory Management
🧠 Storage vs Execution Memory

Every Spark executor has a fixed memory budget split into two pools. Getting the ratio wrong is the #1 cause of mysterious OOM errors and spill-to-disk slowdowns.

PoolControlsDefault
Execution Shuffle buffers, join hash tables, aggregation state. Active during computation. 60% of unified pool
Storage Cached RDDs and DataFrames (.cache(), .persist()). Persists across actions. 40% of unified pool
⚠️
Both pools borrow from each other dynamically. But when execution memory is exhausted during a large shuffle, Spark spills to disk — this slows a job by 10–50×. Check the Stages tab in Spark UI for "Shuffle spill (disk)".
spark-submit — memory config

📐 Core Concepts

Kafka has a small set of primitives. Master these and everything else clicks.

Topic named, ordered, immutable event log Partition parallel unit within a topic (more partitions = more throughput) Offset monotonic position in a partition (like a line number, never decreases) Producer writes events; key determines partition Consumer reads from an offset, tracks position Group N consumers share M partitions (one partition → one consumer at a time) Broker a Kafka server (run 3+ for HA) ISR In-Sync Replicas — who has latest data Retention 7 days by default / size-based
💡
Schema Registry sits alongside Kafka and enforces Avro/Protobuf schemas on every topic. Producers register a schema → get an ID. Consumers fetch by ID → deserialize. This prevents a producer schema change from silently breaking all consumers.
💡
Plain English — CAP Theorem In any distributed system (like Kafka across 3 servers), you can only guarantee two of three properties at once: Consistency (every read sees the latest write), Availability (every request gets a response), and Partition Tolerance (the system works even if servers can't talk to each other). Kafka prioritizes Availability + Partition Tolerance — it keeps accepting writes even if a broker goes offline, then syncs when it's back. This is why you might read a slightly stale message, but you'll never lose data.
🐍 Python Producer + Consumer

Partition Key Strategy
🗝️ The Most Consequential Kafka Decision

Your choice of partition key determines how messages are distributed across partitions — which controls ordering, parallelism, consumer lag, and whether you get data skew. Get this wrong and one partition handles 90% of traffic while the others sit idle.

Key choiceDistributionOrdering guaranteeRisk
customer_idEven (if IDs uniform)All events for a customer in orderLow — ideal for user-event streams
regionUneven — US 10x othersAll events for a region in orderHigh skew — US becomes a bottleneck
null (random)Perfectly evenNone — same entity scatters everywhereFine for logging, bad for joins
event_typeSkewed if one type dominatesAll events of same type in orderBad if "click" is 100x more than "purchase"
💡
Plain English Imagine 8 checkout lanes at a supermarket. A good partition key sends customers evenly across all lanes. A bad one (like "surname A-C") sends half the store to 3 lanes and leaves 5 empty. Those 3 lanes are your hot partitions — they lag behind while the rest sit idle.
⚡ Consumer Lag & Skew — What Goes Wrong

Consumer lag is the gap between the latest message produced and the latest one consumed. A well-partitioned topic keeps lag near zero across all partitions. A skewed one creates a permanent lag mountain on the hot partition.

Kafka CLI — diagnosing partition skew
# Check consumer group lag per partition $ kafka-consumer-groups.sh \ --bootstrap-server kafka:9092 \ --group orders-consumer \ --describe # Healthy: lag balanced across partitions # PARTITION LAG # 0 100 # 1 100 # 2 98 # # Skewed: hot partition can't keep up # PARTITION LAG # 0 700000 <-- HOT # 1 1000 # 2 980
Good keys: high-cardinality IDs (user_id, order_id) that are uniformly distributed.
Bad keys: low-cardinality fields (status, country) or anything with a power-law distribution.
Golden rule: partition count sets your maximum consumer parallelism. Start higher than you think you need — you can't reduce without recreating the topic.
Quick Pick 🗝️ Need message ordering per user → Partition by user_id ⚡ Need max parallelism, no ordering needed → Null partition key (round-robin) 📊 Consumer falling behind (high lag) → Add consumers up to partition count, then increase partitions 🔄 Real-time CDC from Postgres/MySQL → Debezium → Kafka → Bronze
🔥
How it fails: Consumer lag spikes usually mean one of three things: (1) a hot partition from a bad partition key — check lag per partition with kafka-consumer-groups.sh --describe, (2) a consumer that's too slow — profile the processing logic, (3) a rebalance storm — consumers joining/leaving faster than the rebalance timeout, causing progress to reset. Check consumer group state: "Stable" is healthy, "PreparingRebalance" means churn.

Circuit Breakers
⛔ Stop Bad Data Before It Reaches Gold

A circuit breaker is a pattern that automatically halts a pipeline when a data quality check fails — preventing corrupted or incomplete data from flowing downstream into your Gold layer and poisoning dashboards, reports, or ML models. Think of it as the emergency stop button wired to your DQ tests.

💡
Plain English Without a circuit breaker: bad data flows silently from Silver to Gold to dashboard. Finance presents wrong numbers to the board. With a circuit breaker: the pipeline stops the moment a check fails, fires an alert, and keeps the last-known-good Gold layer intact until someone fixes the problem.

Backfill Strategies
⏪ Re-processing Years of Data Without Breaking the Budget

Business logic changes. A revenue calculation bug gets discovered. A new dimension needs to be joined back in time. Suddenly you need to reprocess 3 years of 10TB/day data. Without a strategy, this either costs a fortune, takes weeks, or both.

💡
Plain English Backfilling is like having to repaint every room in a hotel because someone changed the brand color — ideally you have a system to do it floor-by-floor without closing the whole hotel, not all at once.
Partition-by-partition backfill — reprocess one day/week partition at a time. Limits blast radius, allows progress monitoring, and uses minimal concurrent compute. The default approach for large datasets.
Parallel backfill with rate limiting — reprocess N partitions concurrently using Airflow catchup + max_active_runs=4. Faster, but watch cloud costs — set a warehouse size limit or budget alert first.
Blue-green backfill — write to a new table (fct_orders_v2) in parallel while keeping the old one live. Swap the BI view atomically once backfill is validated. Zero downtime, zero dashboard interruption.
Selective backfill — only reprocess affected partitions, not the full history. If a bug only affects orders > $10,000, filter first. Can reduce compute cost by 90%.
Never truncate-and-reload at scale — your Gold table is empty during the load. Dashboards show zeros. Finance calls. Always use the blue-green approach for tables that serve live BI.
Airflow — controlled backfill

📸 Slowly Changing Dimensions (SCDs)

What do you do when a customer moves cities, or a product changes its name? SCDs define the strategy for tracking dimension history.

💡
Plain English — SCD Types Imagine tracking a customer's home city. SCD 1 is a whiteboard — erase the old city, write the new one. Simple, but history is gone. SCD 2 is a passport with entry stamps — a new row per change, with dates, so you know exactly where they lived at any point in time. SCD 1
Overwrite. No history. The row just updates in place. Use for: typo corrections, non-analytical fields. Simple but lossy.
SCD 2
New row per change with valid_from, valid_to, is_current flags. Full history preserved. The default choice for analytics. dbt snapshots automate it.
SCD 3
Add a previous_value column. You keep exactly one prior version. Use when only the last change matters, not full history.
SCD 4
Separate history table. Current record + full audit log table. For compliance-heavy environments. Highest storage cost.

PII Masking Patterns
🛡️ Protecting Sensitive Data in the Silver Layer

The Silver layer is where you clean and conform raw data — and it's also where you apply PII masking and hashing. Never land raw PII in Gold tables that analysts query. The pattern: hash/tokenize in Silver, expose only what's needed in Gold.

Bronze
Raw PII arrives: email = "alice@example.com", phone = "+1-555-0100". Kept as-is — this is the source of truth. Access restricted to data engineers only.
Silver
PII is masked or hashed on write. Analysts never query Silver directly. Pseudonymisation: replace PII with a stable token that links records without exposing the value.
Gold
Only hashed tokens and derived attributes exposed. email_hash instead of email. GDPR erasure = delete from Bronze + invalidate token mapping.
dbt — PII masking in Silver model
-- models/silver/stg_users.sql SELECT user_id, -- Hash PII — stable, pseudonymous, not reversible SHA2(LOWER(TRIM(email)), 256) AS email_hash, SHA2(phone, 256) AS phone_hash, -- Keep non-PII attributes as-is country, signup_date, plan_type, -- Generalise age (k-anonymity technique) FLOOR(age / 10) * 10 AS age_bracket FROM bronze.raw_users
⚠️
Salt your hashes. Unsalted SHA2(email) is reversible via rainbow tables for common email formats. Use SHA2(CONCAT(email, secret_salt)) or a dedicated tokenisation service (AWS KMS + custom logic, AWS Nitro Enclaves, or HashiCorp Vault Transform). Note: AWS Macie is a discovery/classification tool — it finds PII, it does not tokenise it.

Vector Databases — AI / RAG

🧠 What & Why

Traditional databases find exact matches: WHERE city = 'Paris'. Vector databases find semantic neighbors: "find me the 5 documents most similar in meaning to this query." Core use cases: RAG for LLMs, semantic search, recommendation engines, anomaly detection, duplicate record detection.

AI / RAG Engineering
🧠 ETL-to-Vector: The Full Pipeline (Chunking → Embedding → Upserting)

This is the most common DE task in 2025/26 that isn't in any traditional curriculum. Retrieval-Augmented Generation (RAG) systems need fresh, queryable embeddings. Building the pipeline that keeps them up to date is a pure data engineering problem: ingest documents, chunk them intelligently, embed them, upsert to a vector store, and serve them to an LLM.

Step 1 — Chunk

Split documents into overlapping text chunks. Too small = context lost. Too large = embedding quality drops. Overlap between chunks preserves sentence continuity at boundaries.

Python
from langchain.text_splitter import ( RecursiveCharacterTextSplitter ) splitter = RecursiveCharacterTextSplitter( chunk_size=512, # tokens per chunk chunk_overlap=64, # overlap at boundaries separators=["\n\n", "\n", " "] ) chunks = splitter.split_text(document) # Preserve metadata with each chunk docs = [{ "text": chunk, "doc_id": doc_id, "chunk_index": i, "source": source_url, } for i, chunk in enumerate(chunks)]
Step 2 — Embed

Convert each text chunk into a dense vector using an embedding model. Similar meaning = similar vectors. The embedding model choice affects retrieval quality more than anything else.

Python
import openai, numpy as np client = openai.OpenAI() def embed_batch(texts, model="text-embedding-3-small"): # Batch API call — don't embed one at a time resp = client.embeddings.create( input=texts, model=model ) return [r.embedding for r in resp.data] # Process in batches of 100 batch_size = 100 for i in range(0, len(docs), batch_size): batch = docs[i:i+batch_size] vectors = embed_batch([d["text"] for d in batch]) for doc, vec in zip(batch, vectors): doc["embedding"] = vec
Step 3 — Upsert & Query

Write vectors to the vector DB with their metadata. Use upsert (not insert) so re-running the pipeline updates changed documents without duplicating them.

Python — Pinecone
from pinecone import Pinecone pc = Pinecone(api_key="...") index = pc.Index("knowledge-base") # Upsert with metadata for filtered search index.upsert(vectors=[ (f"{doc['doc_id']}-{doc['chunk_index']}", doc["embedding"], {"text": doc["text"], "source": doc["source"]} ) for doc in docs ]) # Query: embed user question → find top-k chunks query_vec = embed_batch([user_question])[0] results = index.query( vector=query_vec, top_k=5, include_metadata=True ) context = "\n".join( r.metadata["text"] for r in results.matches )

Orchestrate with Airflow or Dagster. The ETL-to-Vector pipeline should run on a schedule — daily or on document change events. Track which doc_ids have been embedded (checkpointing) so you only re-embed changed or new documents. Store chunk metadata in Postgres or Delta Lake alongside the vector store for lineage and debugging.

They solve completely different problems and are often used together: OLAP for your dashboards, vector DB for your AI features.

🐋 Multi-Stage Dockerfile

Multi-stage builds produce lean runtime images by separating the build environment (with compilers and build tools) from the runtime image (only what the app needs). This shrinks your image from ~1GB to ~150MB and reduces attack surface.

Dockerfile
# Stage 1 — Build layer (has all dev tools) FROM python:3.11-slim AS builder WORKDIR /build COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # Stage 2 — Lean runtime (ships to prod) FROM python:3.11-slim AS runtime WORKDIR /app # Copy only installed packages from build stage COPY --from=builder /usr/local/lib/python3.11 . COPY ./src . ENV PYTHONUNBUFFERED=1 USER nobody # never run as root in prod ENTRYPOINT ["python", "pipeline.py"]

Best Practices & Anti-Patterns

This is the section you earn after years of production incidents, 3am PagerDuty alerts, and six-figure cloud bills. Every item on the "Do These" list exists because someone learned it the hard way. Every anti-pattern is a real failure mode that has brought down production pipelines. Read this section carefully — not as theory, but as hard-won engineering wisdom distilled into actionable rules you can apply starting today.

What's covered
12 golden rules to follow always
12 anti-patterns that cause incidents
Reasoning behind each rule
✅ Do These — Always
Idempotent pipelines. Re-running must always produce the same result. Use MERGE, partition overwrite, or INSERT OVERWRITE. Plain appends without deduplication will double your data on every backfill.
Immutable raw layer. Raw data, once landed, is never modified, cleaned, or deleted. It's your source of truth for reprocessing. Treat it like an audit log — append-only forever.
Schema-on-write for marts. Enforce explicit schemas with DDL on curated tables. Schema-on-read is fine in the raw zone; marts need strict contracts so downstream consumers don't break silently.
Explicit Spark schemas. Always define StructType manually. inferSchema is slow, fails on all-null columns, breaks on schema drift, and lies when column types are ambiguous.
Partition on query patterns. Partition by date (year/month/day). Never by user_id, UUID, or high-cardinality columns — you'll create millions of tiny files and a metadata nightmare.
Monitor freshness and volume. Alert when MAX(updated_at) is stale OR row count drops >20% vs the 7-day rolling average. Silent data stops are the most dangerous failure mode.
Document lineage. Every column should trace back to a source. Use dbt column descriptions + DataHub auto-lineage. When an incident happens, lineage is your forensics tool.
Surrogate keys in the warehouse. Generate MD5/SHA-based surrogate keys. Never use source PKs — they change, get reused, collide across systems, and sometimes contain PII.
Test before promoting. Run dbt test in CI. Block PR merges on not_null, unique, or relationship failures. Promoting untested data to production is a ticking time bomb.
Write data contracts. Agree on schema + SLAs between producers and consumers upfront — in YAML, enforced in CI/CD. A verbal agreement in Slack does not count as a contract.
Infrastructure as code. Every warehouse, bucket, and IAM role lives in Terraform. Never click-ops in production. Reproducible infra means you can rebuild your entire platform in 30 minutes after a disaster.
Tag PII from day one. Classify sensitive columns in your catalog before data proliferates across 50 tables. Retroactive PII tagging is painful, incomplete, and a compliance liability.
❌ Anti-Patterns — Real Failure Modes
SELECT * in pipelines. Schema drift will silently break downstream consumers when a column is added, renamed, or reordered. Name every column explicitly. Always. Without exception.
Transforming in ingestion. Raw must stay raw. The moment you clean or transform data during ingest, you lose the ability to reprocess it correctly when business logic changes.
TRUNCATE + full reload at scale. Your table is empty during the load window. Downstream dashboards show zeros. BI tools get cached zeros. Finance calls. Use incremental MERGE instead.
Skipping quality checks. "I'll add tests later" = "I'll discover the data corruption in a board meeting." Add quality checks before data ships to any consumer, not after.
Too many small files. Millions of tiny Parquet files create metadata overhead that brings down Spark drivers and Hive metastores. Compact with OPTIMIZE or coalesce() regularly.
Hardcoded credentials. Use Secrets Manager, HashiCorp Vault, or environment injection. One leaked secret committed to git — even briefly — is enough to trigger a security incident and a painful credential rotation across every system.
Monolithic DAGs. A 50-task Airflow DAG means a failure at task 40 reruns tasks 1–39. Split into small, focused DAGs with clear SLA boundaries and explicit dependencies.
Ignoring shuffle cost in Spark. Wide transforms (groupBy, join) shuffle data across the network. One unoptimized join on 10B rows can take hours. Profile in the Spark UI (port 4040) before shipping.
Function on partition column. WHERE YEAR(ts) = 2024 wraps the partition column in a function, defeating partition pruning and causing a full table scan. Use range filters on the raw column.
No cost guardrails. An unbounded Snowflake warehouse with no auto-suspend and no resource monitor is a cloud billing time bomb. Set guardrails on day one, not after the first surprise invoice.
Central team bottleneck. If every domain must request a pipeline from the central DE team, you will not scale past 50 analysts. Invest in self-serve tooling and domain enablement, or adopt Data Mesh principles.
No data contracts. Producers silently renaming a column is the #1 cause of downstream pipeline failures. If there's no contract, there's no warning. Contract first, always.

A real one, with someone's phone ringing at 3am. The "Do These" list isn't bureaucracy — it's the accumulated scar tissue of hundreds of engineers who learned these lessons the expensive way.
